\chapter{Implementation and Optimization}
\label{chap:implementation}

This chapter shows how the validated logical plan from
Chapter~\ref{chap:methodology} becomes a running engine: how a pattern
is compiled into a finite automaton, how that automaton is executed
over ordered Pandas partitions, and how a layer of compiled fast paths
makes the engine fast without changing its results.
Section~\ref{sec:meth-automata} covers automata compilation,
Section~\ref{sec:meth-pandas} the DataFrame execution model, and
Section~\ref{sec:bg-performance} the optimization layer.

% ----------------------------------------------------------------
\section{Pattern Compilation with Finite Automata}
\label{sec:meth-automata}
% ----------------------------------------------------------------

Row-pattern matching is naturally a sequence-recognition problem: if
each data row is labelled with the pattern variables it satisfies, the
sequence of rows becomes a string over the alphabet of pattern
variables, and deciding whether the pattern accepts that string is
exactly what a finite automaton does~\cite{HopcroftUllman}.  The engine
therefore compiles every pattern into an automaton before matching
begins.

\paragraph{From pattern to NFA.}
A \emph{non-deterministic finite automaton} (NFA) can be in several
states at once and can take \emph{$\varepsilon$-transitions} that
consume no input, which makes it a natural fit for alternation and
optional constructs.  Figure~\ref{fig:meth-nfa} shows the NFA for the
pattern $A^{+}\,B?$.  The engine builds it by Thompson's
construction~\cite{Thompson1968,DragonBook}, which recurses over the
pattern: a variable becomes a two-state fragment, concatenation joins
one fragment's end to the next one's start, alternation forks and
merges the branches through $\varepsilon$-edges, and a quantifier adds
the appropriate skip and loop $\varepsilon$-edges.  The implementation
follows this scheme but adds wrapper states, predicate metadata,
transition priorities, and guards for SQL-specific constructs, so the
final state count depends on the constructs used rather than on the
textbook $2n$ bound.

\paragraph{Tokenizing the pattern.}
Construction starts from the raw pattern string, which a single
left-to-right scan turns into a flat list of tokens: variables,
alternation, grouping, anchors, exclusions, and \texttt{PERMUTE}.  Any
trailing quantifier (\texttt{*}, \texttt{+}, \texttt{?}, or
\texttt{\{$n$,$m$\}}, optionally made reluctant by a further
\texttt{?}) is attached to the token it modifies and normalised to a
$(\mathit{min},\,\mathit{max},\,\mathit{greedy})$ triple, so the NFA
builder never has to re-parse quantifier syntax.

\paragraph{Building the NFA.}
A visitor then walks the token list and emits the matching NFA fragment
for each token, mirroring Thompson's recursion over a flat list rather
than an explicit tree.  A variable becomes a two-state fragment whose
transition carries the callable compiled from that variable's
\texttt{DEFINE} predicate.  A quantifier wraps a fragment in envelope
states with the right skip and loop $\varepsilon$-edges; for a greedy
quantifier the loop-back edge is given higher priority than the exit
edge, so the $\varepsilon$-closure extends the match before ending it.
\texttt{PERMUTE}$(v_1,\ldots,v_n)$ expands to the union of all $n!$
orderings, protected by a complexity guard---patterns with more than
eight variables, or an estimated fifty thousand branches, are rejected
or routed to a conservative fallback.  An exclusion \texttt{\{-\ \ldots
\ -\}} builds its sub-pattern normally but marks the states so the
matcher later drops those rows from the output.  Compiled NFAs are
cached, checked for accept-state reachability, and guarded for
thread-safe reuse.

\paragraph{From NFA to DFA.}
The NFA is finally converted to a \emph{deterministic} finite automaton
(DFA) by subset construction, in which each DFA state is a set of NFA
states.  This removes $\varepsilon$-transitions and leaves at most one
successor per symbol, so a partition can be scanned row by row with no
backtracking.  Subset construction is exponential in the worst case,
but \texttt{MATCH\_RECOGNIZE} patterns are structurally constrained and
yield small DFAs in practice.  Crucially, the transition priorities are
propagated through the construction, so the DFA inherits the
\emph{greedy, leftmost-first} preference that SQL:2016 requires;
unreachable states are then removed and compatible states merged,
though this is a practical optimisation rather than full canonical
minimisation.

\section{Pandas DataFrame Execution Model}
\label{sec:meth-pandas}
% ----------------------------------------------------------------

Pandas~\cite{McKinney2010} stores each DataFrame column as an
array, which makes it well suited to the row-by-row scan that matching
needs and to the vectorised fast paths of
Section~\ref{sec:bg-performance}.  Execution turns a validated clause
and a compiled DFA into output rows through six ordered steps:
partition, order, row classification, DFA simulation, measure
evaluation, and output assembly.  Figure~\ref{fig:exec-pipeline} places
these steps within the overall architecture of
Figure~\ref{fig:system-architecture}: the SQL query flows down
phases~1--4 into a compiled DFA, and the shaded execution phase then
combines that matcher with the input DataFrame to produce the result.

\paragraph{Partition, order, and classify.}
Execution begins by splitting the DataFrame into independent
partitions with \texttt{groupby()} on the \texttt{PARTITION BY} columns
(a single global partition when there is none) and sorting each by
\texttt{ORDER BY}.  Partitions share no state and can be processed
independently, so the rest of this section follows one sorted
partition.  Each \texttt{DEFINE} predicate is compiled once into a
callable that receives a row and a \emph{row context} holding the
sorted partition, the current variable bindings, and the
\texttt{SUBSET} definitions; variables that appear in \texttt{PATTERN}
but not in \texttt{DEFINE} carry the implicit always-true predicate
fixed during logical planning
(Chapter~\ref{chap:methodology})~\cite{trino2022}.  Predicates are evaluated lazily as the
matcher attempts each transition, and those that reference running
aggregates or \texttt{CLASSIFIER()} bypass the condition cache because
their value depends on the evolving match rather than on the row alone.

\paragraph{DFA simulation.}
The core loop walks the DFA from its start state over the sorted rows.
At each row it evaluates the \texttt{DEFINE} predicates on the current
state's outgoing transitions, keeps those that hold, and takes the
highest-priority one; reaching an accepting state records a candidate
match, and the walk continues to seek the longest greedy match before
stopping.  When several transitions fire at once, priority prefers
transitions into accepting states, then self-loops (the greedy
extension of a quantifier such as \texttt{A+}), then---for
cross-variable predicates such as \texttt{B.price > A.price}---the
prerequisite variable before the dependent one, with any remaining ties
broken by leftmost alternation order.  Each successful match yields
\emph{bindings} that map every variable to its list of matched row
indices; these later drive the measures and navigation functions.  Rows
inside an exclusion participate in matching but are omitted from the
output.

\subsection{Exact Search for State-Dependent Conditions}
\label{subsec:meth-exec-backtracking}
% ................................

The greedy walk assumes a predicate's outcome depends only on the row
under test and previously fixed bindings.  That assumption breaks for
\emph{state-dependent} conditions---aggregates or \texttt{CLASSIFIER()}
in \texttt{DEFINE}, where accepting a row under one label changes the
value of predicates tested later---and for patterns whose preference
order can only be established by trying alternatives (a quantified
group containing an alternation with a multi-element branch).  For
these queries a gate function routes the query to a dedicated
searcher that performs a preference-ordered depth-first search over the
parsed pattern: at each position it tentatively assigns the row to the
highest-preference viable alternative, re-evaluates the affected
conditions under that assignment, and backtracks when a later condition
fails.  A step budget bounds the worst-case exponential search; within
that budget the searcher reproduces Trino's tentative-label SQL:2016
semantics exactly---verified on the set-partitioning and
balancing-sums queries that cannot be matched greedily---while the
common row-local patterns keep the fast greedy path.

\subsection{Skip Modes, Output Modes, and Measures}
\label{subsec:meth-exec-output}
% ................................

After recording a match the engine must decide where to resume, which
controls whether matches can overlap.  SQL:2016 defines four
\texttt{AFTER MATCH SKIP} modes, summarised with their next-start
computation in Table~\ref{tab:skip-modes}.  A monotonic-progress
invariant---the next start position is always strictly greater than the
current one---together with cycle detection for \textsc{To Next Row}
prevents infinite loops.

\begin{table}[htbp]
  \centering
  \caption{The four \texttt{AFTER MATCH SKIP} modes and the
    resulting next scan position}
  \label{tab:skip-modes}
  \begin{tabular}{lp{9cm}}
    \toprule
    \textbf{Mode} & \textbf{Next-start computation} \\
    \midrule
    \textsc{Past Last Row} (default) &
      $i \leftarrow \mathit{best.end} + 1$. No overlap is possible;
      the scan resumes immediately after the last matched row. \\[4pt]
    \textsc{To Next Row} &
      $i \leftarrow \mathit{best.start} + 1$. Overlapping matches
      are produced because only a single row is skipped regardless
      of match length. \\[4pt]
    \textsc{To First} $V$ &
      $i \leftarrow \min(\mathit{bindings}[V])$. The scan
      resumes at the \emph{first} row assigned to variable $V$
      ($V$ may also be a \texttt{SUBSET} alias).  Skipping to the
      first row of the match itself is reported as a runtime error,
      since it would loop forever. \\[4pt]
    \textsc{To Last} $V$ &
      $i \leftarrow \max(\mathit{bindings}[V])$. The scan
      resumes at the \emph{last} row assigned to variable $V$ (or
      \texttt{SUBSET} alias); if $V$ matched no row, a runtime error
      is reported. \\
    \bottomrule
  \end{tabular}
\end{table}

The engine supports the two output-row multiplicities of SQL row-pattern
recognition~\cite{trino2022}, summarised in
Table~\ref{tab:rows-per-match}.  \texttt{ONE ROW PER MATCH} emits one
row per match, evaluating each measure once with \textsc{Final}
semantics (all bindings visible).  \texttt{ALL ROWS PER MATCH} emits
one row per matched row, tagging each with its \texttt{CLASSIFIER}
(the variable it matched) and evaluating measures under
\textsc{Running} or \textsc{Final} semantics per function; a
\textsc{With Unmatched Rows} variant additionally emits every input row,
with \texttt{NULL} measures for unmatched ones.

\begin{table}[htbp]
  \centering
  \caption{Output modes and their row-count and measure-semantics
    behavior}
  \label{tab:rows-per-match}
  \begin{tabular}{>{\raggedright\arraybackslash}p{3.2cm}%
    >{\raggedright\arraybackslash}p{2.3cm}p{6.6cm}}
    \toprule
    \textbf{Mode} & \textbf{Rows out} & \textbf{Measure evaluation context} \\
    \midrule
    \textsc{One Row Per Match}
      & 1 per match
      & \textsc{Final}: measures see the complete, finalized
        variable bindings.  Navigation functions
        (\texttt{FIRST}, \texttt{LAST}) operate over the full
        matched span. \\[6pt]
    \textsc{All Rows Per Match}
      & 1 per matched row
      & Implementation default per function: \texttt{LAST} uses
        \textsc{Running}; \texttt{FIRST}/\texttt{PREV}/\texttt{NEXT}
        use \textsc{Final}; \texttt{COUNT}, \texttt{SUM}, and
        \texttt{ARRAY\_AGG} use \textsc{Running}; other aggregate
        functions default to \textsc{Final}.
        Explicit \texttt{RUNNING}/\texttt{FINAL} keywords override
        the default. \\[6pt]
    \textsc{All Rows \ldots With Unmatched}
      & 1 per every input row
      & Matched rows carry measure values; unmatched rows emit
        \texttt{NULL} for all measure columns. \\
    \bottomrule
  \end{tabular}
\end{table}

Measures are compiled to callables in the same way as the conditions.
Navigation functions (\texttt{FIRST}, \texttt{LAST}, \texttt{PREV},
\texttt{NEXT}) resolve to bounds-checked index lookups into the
partition---an out-of-range reference returns \texttt{NULL} per
SQL:2016---while aggregates (\texttt{COUNT}, \texttt{SUM}, \texttt{AVG},
and the others) range over the bound row sets.  The
\textsc{Running}/\textsc{Final} distinction matters most under
\texttt{ALL ROWS PER MATCH}: a running measure evaluated for output row
$j$ sees only the rows up to $j$ (so a \texttt{SUM} builds up
cumulatively), whereas a final measure always sees the complete match.

\section{Performance Optimization Techniques}
\label{sec:bg-performance}
% ----------------------------------------------------------------

The optimisations fall into two groups.  The first is a set of
\emph{compiled fast paths} in the matcher: per-query code paths chosen
after inspecting the pattern and its \texttt{DEFINE} predicates, each
falling back to the general evaluator when it meets a construct it does
not cover.  Every fast path must be output-identical to the general
one, checked by differential tests that compare full result DataFrames.
The second group is \emph{auxiliary infrastructure}---caching, memory
management, and scheduling.  Table~\ref{tab:perf-summary} lists every
technique with its trigger and benefit; the paragraphs below cover the
fast paths, where most of the speed-up comes from.

\paragraph{Vectorised row-local conditions.}
A \texttt{DEFINE} predicate built only from the current row's columns
and \emph{physical} navigation (\texttt{PREV}/\texttt{NEXT}, which
address row $i\mp k$ regardless of variable assignment) does not depend
on the match state.  So it is evaluated once per partition as a column
expression: navigation becomes a Pandas \texttt{shift($k$)},
comparisons become vectorised Boolean arrays, and the DFA reads
precomputed truth values instead of calling a Python closure per row.
Predicates using \texttt{FIRST}/\texttt{LAST}, \texttt{CLASSIFIER},
aggregates, or cross-variable references keep the scalar evaluator.

\paragraph{Row-local DFA decision tables.}
When every predicate is vectorised, each DFA state's winning transition
is a pure function of the row index.  For each state the engine
precomputes a byte array over the partition whose $i$-th entry encodes
the highest-priority transition that holds at row $i$, so the
simulation loop is just indexing \texttt{bytes} objects at C speed.
Assignments are stored as $(\mathit{var}, \mathit{start},
\mathit{stop})$ run segments, not per-row appends, and expanded to rows
only for the accepted match---so failed walks never materialise per-row
structures.

\paragraph{Linear-pattern fast path and fused driver.}
A pattern that is a plain concatenation of quantified variables (say
\texttt{A\{1,5\} B* C+}) uses vectorised run-length tables---for each
row, how many consecutive rows satisfy a variable---searched with a
memo of failed \emph{(token, position)} states shared across all start
positions, turning repeated backtracking into table lookups.  On top of
that, when a query uses \texttt{ONE ROW PER MATCH} with
\texttt{SKIP PAST LAST ROW}, no anchors, and a pattern the compiled
matchers cover, a fused driver enumerates matches and emits each output
row straight from its run segments through measure closures compiled
once per query---never building the intermediate per-match dictionaries
or row context of the general path.

\paragraph{Compiled measure plans and prefix aggregation.}
Measures follow the same compile-once idea: column arrays, null masks,
and case-insensitive variable keys are resolved once per partition
pass, not once per match, and semantics analysis is hoisted out of the
per-row loop.  Arithmetic over \emph{running} aggregates (for example
\texttt{RUNNING sum(A.value * A.score) / sum(A.score)}), which naive
per-row evaluation would make quadratic, is compiled into a single
$O(n)$ prefix pass over the match rows.

\paragraph{Auxiliary infrastructure.}
Four supporting mechanisms sit behind the fast paths.  A thread-safe
\emph{LRU pattern cache} maps a hash of the pattern and its
\texttt{DEFINE} predicates to the compiled NFA/DFA, so repeated queries
never recompile.  Because partitions are independent, a parallel-aware
scheduling layer keeps a boundary for future concurrent execution while
the current path runs partitions one at a time.  Memory management is
partition-oriented---one partition at a time, RSS watched with
\texttt{psutil}, garbage collection triggered before limits---and an
object pool reuses hot objects on long queries.  Finally, for large
partitions the engine enlarges its condition cache and pre-warms common
predicate paths on a sample of rows before the full scan.

\section{Chapter Summary}
\label{sec:impl-summary}
% ----------------------------------------------------------------

This chapter described how a pattern is compiled into an NFA and then a
DFA, how that DFA is executed over ordered Pandas partitions---with
after-match skip, output modes, and measure evaluation---and how
state-dependent conditions are handled by an exact backtracking
searcher when the greedy walk does not suffice.  It then presented the
optimisation layer: a set of compiled fast paths (vectorised row-local
conditions, byte-encoded DFA decision tables, a linear-pattern search
with a shared failure memo, a fused \texttt{ONE ROW PER MATCH} driver,
and compiled measure plans with an $O(n)$ running-aggregate pass),
backed by pattern caching, parallel-aware scheduling, and
partition-oriented memory management.  Because every fast path is
required to produce output identical to the general evaluator, these
optimisations are orthogonal to the matching semantics and can be
evaluated on their own merits.  Chapter~\ref{chap:evaluation} does
exactly that, measuring correctness, throughput, memory behaviour, and
performance across the selected pattern families and dataset sizes.
