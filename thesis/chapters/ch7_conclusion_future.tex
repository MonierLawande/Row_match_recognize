\section{Discussion}
\label{sec:discussion}
% ----------------------------------------------------------------

The results show a practical subset of SQL row-pattern recognition can
move into \texttt{pandas} without giving up the declarative shape of
\texttt{MATCH\_\allowbreak{}RECOGNIZE}.  For the reported reference
queries, the engine returned the same rows and measure values as
Trino~473, Oracle, and hand-written \texttt{pandas}---including
partition-local ordering, implicit \texttt{DEFINE} behavior,
\texttt{PREV} navigation, \texttt{MATCH\_NUMBER()}, and measure
materialization.  That is evidence the implemented subset preserves the
evaluated semantics, not a claim of parity with every commercial SQL
feature.

The comparison with idiomatic \texttt{pandas} matters too.  An
equivalent procedural version needs explicit \texttt{groupby},
\texttt{shift}/\texttt{diff}, candidate filtering, index bookkeeping,
and manual skip handling; the SQL form says the same thing in one
statement.  That lowers developer effort, makes the pattern easier to
read, and travels between notebook analysis and SQL workflows---as long
as the query stays inside the supported subset.

Performance backs the approach for medium-scale in-memory work.
Chapter~\ref{chap:evaluation} reported 30 pattern--size cells (each the
mean of twenty runs after five warmups) over five families and six
sizes, 100\,K to 2.22\,M rows.  Throughput averaged
4\,976\,348~rows/s and held steady as input grew; the variation came
from pattern shape---simple sequences and bounded quantifiers were
fastest, \texttt{complex\_nested} slowest.  Under one shared budget the
engine had the lowest average time of the three (0.20\,s versus
Oracle's 1.04\,s and a warm Trino's 1.98\,s) and won every one of the
30 cells---roughly 5.5--14.5$\times$ over Trino and 2.8--6.9$\times$
over Oracle.  The reason is narrow specialization: no query planner or
distributed layer, fully in-process execution over
\texttt{pandas}/NumPy arrays, and compiled fast paths that turn per-row
work into array operations.  It holds only for the supported subset and
the single-node in-memory setting.  A separate scalability test then
pushed all three to the 227.9\,M-row Amazon Reviews~2023
corpus---about $100\times$ larger.  The engine's time and memory stayed
near-linear and it kept a $6$--$11\times$ lead; Oracle~EE also reached
the full corpus, while Trino's hardest pattern ran out of the time
budget.  The price is memory: the engine holds the scanned data in RAM,
so its per-query peak is the highest of the three and grows with the
data---below 300\,MB at 2.22\,M rows, tens of gigabytes at the full
corpus.

Architecturally, the system keeps parsing, semantic validation,
automata construction, matching, and materialization separate.  It uses
an ANTLR4 parser on the Trino grammar, normalized internal forms,
NFA/DFA construction, a \texttt{pandas} execution layer, and measure
evaluation.  This modularity makes it easier to validate and to
extend---improving compilation, adding a backend, or extending measures
needs no redesign.

The optimization layer helps but should be read precisely.  It includes
LRU-style pattern caching, bounded memory management, condition-cache
pre-warming, and a parallel-aware scheduling abstraction.  Still, this
evaluation measures the implemented \texttt{pandas} execution path, not
a distributed or broadly parallel SQL engine---Trino and Oracle bring
mature optimizers, wider dialect coverage, and execution infrastructure
that stay outside this prototype.

The central finding is scoped but real: SQL row-pattern semantics can
be compiled into a finite-automata engine and run directly over
DataFrames, with correct results on the evaluated scenarios and
predictable performance across the measured range.

The evidence carries three threats to validity.  \emph{Internal
validity} rests on correct reference baselines and careful
normalization of case-only column differences between SQL and
\texttt{pandas}.  \emph{Construct validity} rests on the metrics
capturing what they should: row-level equality for correctness,
throughput for speed, line count for one facet of developer effort.
\emph{External validity} is bounded by one workstation, one main
real-world dataset, and a limited size range---and by running Trino and
Oracle at default container settings rather than expert-tuned, so a
tuned deployment could narrow the gap.  These do not undo the results;
they set the conditions for generalizing them.

\section{Limitations}
\label{sec:conc-limitations}
% ----------------------------------------------------------------

A few limitations bound how far the results generalize.

\textbf{Supported subset, not exhaustive SQL.}
The engine targets a substantial but bounded slice of
\texttt{MATCH\_\allowbreak{}RECOGNIZE}: \texttt{PARTITION BY},
\texttt{ORDER BY}, \texttt{PATTERN}, \texttt{DEFINE}, selected
\texttt{MEASURES}, supported skip and output modes, navigation,
subsets, exclusion blocks, anchors, and \texttt{PERMUTE}.  It does not
claim full SQL:2016 coverage or parity with Trino or Oracle;
vendor-specific extensions and rare edge cases stay outside the
validated scope.

\textbf{Complex pattern and quantifier interactions.}
The automata design handles ordinary sequences, alternation, grouping,
bounded repetition, and common quantifiers.  Deeply nested expressions
with several greedy or unbounded quantifiers can blow up the state and
transition count, and \texttt{PERMUTE} is especially costly because its
semantics are factorial---so the engine leans on complexity guards and
conservative fallback for large cases.

\textbf{Measure and aggregate scope.}
The measure evaluator supports common functions---\texttt{SUM},
\texttt{COUNT}, \texttt{MIN}, \texttt{MAX}, \texttt{AVG}, statistical
and collection/string aggregates, conditional aggregates, and supported
\texttt{RUNNING}/\texttt{FINAL} forms.  Arbitrary user-defined
aggregates and every possible expression combination remain out of
scope---a natural limit of an in-memory prototype with its own
expression evaluator rather than a full SQL engine.

\textbf{Performance boundaries.}
The benchmark shows predictable behavior from 100\,K to 2.22\,M rows on
one workstation.  Larger datasets, wider schemas, more partitions, and
highly ambiguous patterns may expose bottlenecks in Python execution,
\texttt{pandas} grouping, row-context construction, and predicate
evaluation.  Native SQL engines and compiled columnar systems can still
give higher throughput at large scale.

\textbf{Memory behavior.}
Memory stayed bounded in the benchmark and query memory grows with
input as expected.  But readings near the floor are quantized to each
runtime's allocation granularity (memory-pool chunks for the databases,
resident pages for Python), and small values shift with GC timing and
result materialization.  So the memory numbers are empirical
measurements for the tested workloads, not a formal upper bound.

\textbf{Optimizer integration.}
The engine runs independently of database optimizers.  It does no
cost-based plan selection, predicate pushdown into external storage,
join reordering, or distributed scheduling, so some plan-level
optimizations that databases offer are simply absent here.

\section{Conclusion}
\label{sec:conclusion}
% ----------------------------------------------------------------

This thesis presented a SQL-in-\texttt{pandas} implementation that runs
a supported subset of \texttt{MATCH\_\allowbreak{}RECOGNIZE} over
DataFrames.  It contributes a Trino-syntax parsing and validation
pipeline, an internal representation of row-pattern queries,
finite-automata pattern compilation, a \texttt{pandas} matcher, measure
evaluation, and optimization support through caching and memory-aware
execution.

The evaluation gives three conclusions.  First, the selected
correctness scenarios matched Trino, Oracle, and hand-written
\texttt{pandas}.  Second, the declarative form cuts the procedural
Python needed for sequential pattern analysis substantially.  Third,
the Amazon UK Products benchmark shows stable performance across 30
cells from 100\,K to 2.22\,M rows---4\,976\,348~rows/s average, the
lowest average time of the three under one budget (0.20\,s versus
1.04\,s Oracle and 1.98\,s warm Trino), and bounded memory over the
range.

The approach is a bridge between SQL row-pattern
recognition and DataFrame analytics.  It does not replace mature SQL
engines; it brings a useful slice of their row-pattern power into
Python workflows where the data already lives in \texttt{pandas}.

\section{Future Work}
\label{sec:conc-future}
% ----------------------------------------------------------------

Five directions would extend the work.

\textbf{Broader SQL and dialect coverage.}
The most direct step is wider SQL:2016 and vendor coverage: more
navigation combinations, broader expressions, richer aggregates, fuller
\texttt{RUNNING}/\texttt{FINAL} semantics, and a larger conformance
suite drawn from Trino, Oracle, and the standard.

\textbf{Parallel and distributed execution.}
Independent partitions are a natural unit of parallel work.  The
existing scheduling abstraction could drive real concurrency while
keeping deterministic ordering and skip semantics.  A further step is
distributed execution on Dask or Spark, which needs boundary-aware
state for matches that span chunks.

\textbf{Backend abstraction and columnar acceleration.}
The path is \texttt{pandas}-centered today.  A backend abstraction
could target Polars or Apache Arrow, pushing more predicate evaluation
into compiled vectorized kernels and cutting Python-level iteration.
Cython, Numba, or compiled predicates could help further, with careful
validation to preserve null handling, navigation, and deterministic
output.

\textbf{Optimization and diagnostics.}
The automata pipeline can gain from stronger state pruning, guard
pushdown, better selectivity estimates, and smarter handling of
high-ambiguity patterns.  Diagnostics matter too: explain plans,
NFA/DFA visualizations, match traces, and per-pattern counters would
make debugging and tuning far easier.

\textbf{Usability and integration.}
Clearer error messages, notebook-friendly visualizations, reusable test
fixtures, and integration with existing DataFrame query interfaces
would make the engine easier to adopt in exploratory analysis and
teaching.

\section{Chapter Summary}
\label{sec:conc-summary}
% ----------------------------------------------------------------

This chapter drew the findings together in a scoped context.  The
implementation supports a useful subset of
\texttt{MATCH\_\allowbreak{}RECOGNIZE} over \texttt{pandas}, with
correctness evidence from selected reference queries and performance
evidence from a benchmark of twenty measured runs per cell over
medium-scale DataFrames.

It also named the main limits---bounded SQL coverage, complex-pattern
growth, expression-evaluation scope, large-scale performance, memory
variability, and limited optimizer integration---and set future work
against them: broader conformance testing, parallel and distributed
execution, backend abstraction, optimization, and better tooling.
