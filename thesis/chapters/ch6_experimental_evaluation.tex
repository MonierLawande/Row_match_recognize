\chapter{Experimental Evaluation}
\label{chap:evaluation}

This chapter evaluates the implementation on five criteria:
correctness against Trino and Oracle, SQL:2016 feature coverage,
performance, scalability, and usability.
Section~\ref{sec:eval-methodology} defines the methodology,
Section~\ref{sec:eval-dataset} the datasets and patterns,
Section~\ref{sec:eval-correctness} validates the concept example by
example, Section~\ref{sec:eval-coverage} reports coverage and
complexity results, and Section~\ref{sec:eval-performance} presents
the cross-system performance analysis.

% ................................
\subsection{Evaluation Criteria and Metrics}
\label{subsec:eval-metrics}
% ................................

\paragraph{Correctness.}
The result DataFrame is compared against two independent SQL
references, Trino~473 and Oracle~XE~21c, by exact row-level equality
(same rows, values, and order where deterministic).  This covers the
two small synthetic tables---\texttt{orders} (Table~\ref{tab:orders})
and \texttt{stock\_price} (Table~\ref{tab:stock_price})---and the 30
pattern--size cells of the Amazon benchmark
(Section~\ref{subsec:eval-cross-system}).  The hand-written Pandas
baseline is not a correctness oracle; it appears only in the usability
dimension (Section~\ref{subsec:eval-usability}).

\paragraph{SQL feature coverage.}
Two views are recorded.  \emph{Feature coverage} marks each main
\texttt{MATCH\_\allowbreak{}RECOGNIZE} clause---partitioning,
ordering, \texttt{PATTERN}, \texttt{DEFINE}, \texttt{MEASURES},
\texttt{SUBSET}, the output modes, skip policies, and navigation
functions~\cite{iso2016,ISO2021,monierashraf2025rowmatch}---as
\emph{supported}, \emph{partially supported}, or \emph{unsupported}
against the reference tests (Table~\ref{tab:feature_coverage}).
\emph{Pattern-type coverage} groups the test cases by row-pattern
type (simple sequences, quantified repetitions, trends, recovery,
alternation, exclusions, anchors, and \texttt{PERMUTE}); a type is
covered when at least one case matches the reference output
(Table~\ref{tab:pattern_type_coverage}).

\paragraph{Performance.}
Execution time is the client-observed end-to-end latency per
pattern--size run on the Amazon UK Products dataset
(Section~\ref{subsec:eval-real-dataset}): five pattern families, six
sizes, result transfer included.  Each cell runs five warmups (the
JVM-based Trino needs them) and twenty measured runs, each preceded
by a five-second idle window that sets the pre-query memory baseline.
Outliers are removed per metric with the $1.5\times$IQR rule; cells
report the mean of the retained runs with its standard deviation.
Throughput is dataset size divided by mean time.  The two memory
metrics---one instrument for all three systems---are defined in
Section~\ref{subsec:eval-memory}.

\paragraph{Scalability.}
Mean execution time is regressed against row count across the six
sizes; the $R^2$ per pattern family shows how close growth is to
linear.

\paragraph{Usability (developer effort).}
For each worked example, the line count of the idiomatic Pandas
solution is compared with the single
\texttt{MATCH\_\allowbreak{}RECOGNIZE} statement
(Table~\ref{tab:loc_comparison}); the reduction measures developer
effort~\cite{iso2016,ISO2021,melton2017sqlrpr,Python_UDF}.

\subsection{Environment and Configuration}
\label{subsec:eval-environment}
% ................................

The experiments ran on a workstation with an Intel Core i7-11800H
processor and 64\,GB of RAM under Ubuntu 24.04.4 LTS.  For the
cross-system benchmark, each system was restricted to one CPU core and
a 32\,GB memory budget; the enforcement mechanisms are detailed in
Section~\ref{subsec:eval-complexity}.  The scalability stress test
(Section~\ref{subsec:eval-stress}) instead uses a 58\,GB budget and, to
reach hundreds of millions of rows on equal terms, replaces the two
database deployments: Trino reads from a disk-backed Hive/Parquet
connector rather than its in-memory connector, and Oracle runs as
21c~Enterprise Edition rather than the resource-capped free XE
edition.  All database containers run on the host's native Docker
daemon so that container limits map directly to host control groups,
the same instrument used for the engine.  The software stack comprised
the two baseline SQL engines---Trino~473 and Oracle~21c, each run as a
Docker container---the Java~11.0.25 LTS and Python~3.12.7 runtimes, and
the core Python libraries \texttt{pandas}~2.3.3, \texttt{numpy}~2.1.3,
\texttt{antlr4-python3-\allowbreak{}runtime}~4.13.2,
\texttt{antlr4-\allowbreak{}tools}~0.2.1, and \texttt{pytest}~8.3.4.
The implementation itself uses ANTLR4 to parse Trino SQL,
\texttt{pandas} for DataFrame management, and custom AST/NFA/DFA
modules for pattern detection (Chapter~\ref{chap:methodology}).

Unless stated otherwise, all experiments use this configuration.
For correctness comparisons, identical data are loaded into all three
systems and result sets are compared by row order and measure values
(after normalizing case-only column-name differences).

\subsection{Datasets}
\label{subsec:eval-dataset-orders}
% ................................

Four datasets are used: two small synthetic tables that can be
verified by hand (\texttt{orders} and \texttt{stock\_price}), the
real-world Amazon UK Products dataset (up to 2.2\,million rows) for the
cross-system timing and memory benchmark, and the Amazon Reviews~2023
corpus (up to 227.9\,million rows) for the scalability test
(Section~\ref{subsec:eval-stress}).

\paragraph{Dataset 1: the \texttt{orders} table.}
The \texttt{orders} table supports the first worked example.  It
comprises three attributes~\cite{trino2021row}: \texttt{customer\_id}
(\texttt{VARCHAR}), which identifies the customer (two distinct
values, \texttt{cust\_1} and \texttt{cust\_2}); \texttt{order\_date}
(\texttt{DATE}), the date the order was placed (ranging from
2020-05-11 to 2020-05-18); and \texttt{price} (\texttt{INTEGER}), the
purchase amount on that date.  Each customer appears multiple times in
chronological order---\texttt{cust\_1} spans five dates and
\texttt{cust\_2} spans three---which enables focused validation of
partitioning, ordering, and pattern detection (e.g.\ sequences of
rising or falling prices per customer).  Table~\ref{tab:orders} shows
the eight records used.

\paragraph{Dataset 3: the Amazon UK Products dataset.}
The Amazon UK Products dataset~\cite{asaniczka2023amazon} holds
roughly 2.2\,million product records from Amazon's UK marketplace as
one CSV file.  The benchmark uses four of its columns:
\texttt{stars} (average rating, 1--5), \texttt{reviews} (review
count), \texttt{price} (GBP), and \texttt{categoryName}; the
remaining columns (IDs, titles, URLs, flags) are not needed.  It
supplies the data volume for the performance benchmark; the derived
\texttt{category} column and the six dataset sizes are described in
Section~\ref{subsec:eval-pattern-defs}.

\paragraph{Dataset 4: the Amazon Reviews 2023 corpus.}
The scalability test (Section~\ref{subsec:eval-stress}) needs far more
rows than the product tables provide, so it uses the Amazon
Reviews~2023 corpus~\cite{hou2024amazonreviews} from the McAuley Lab at
UC~San~Diego---roughly 571~million genuine review events across product
categories.  Only the pattern-relevant \texttt{rating\_only} files are
taken (user, item, rating, timestamp; no review text, images, or
product metadata), and five categories are merged in global timestamp
order into nested prefixes of 5\,M to 227.9\,M rows.  Each review's
$1$--$5$ rating is mapped to the same A--E \texttt{category} alphabet as
the benchmark, so the five pattern families carry over unchanged.
Unlike a tiled copy of a smaller table, every row is a distinct real
event, which makes this the more honest test of scaling behaviour.

\subsection{Example 1: Dip Pattern on the \texttt{orders} Dataset}
\label{subsec:eval-dip}
% ................................

The first example detects a `dip'---three consecutive orders with
successively lower prices---on the \texttt{orders} dataset
(Dataset~1, Table~\ref{tab:orders}).  The same detection is expressed
three ways---as a Trino SQL query (Listing~\ref{lst:orders-trino}), as
a call to the proposed engine (Listing~\ref{lst:orders-mr}), and as a
hand-written Python/pandas UDF (Listing~\ref{lst:orders-udf})---and
Table~\ref{tab:sql-match-recognize-examples} confirms that all three,
together with Oracle, return the identical match.

\begin{lstlisting}[basicstyle=\ttfamily\small,
  backgroundcolor=\color{gray!10},
  caption={[Trino SQL for the dip pattern]Trino SQL: create and load the \texttt{orders} table, then detect the dip pattern with \texttt{MATCH\_\allowbreak{}RECOGNIZE}.},
  label={lst:orders-trino}]
CREATE TABLE memory.default.orders (
    customer_id VARCHAR,
    order_date  DATE,
    price       INTEGER
);

INSERT INTO memory.default.orders VALUES
    ('cust_1', DATE '2020-05-11', 100),
    ('cust_1', DATE '2020-05-12', 200),
    ('cust_2', DATE '2020-05-13',   8),
    ('cust_1', DATE '2020-05-14', 100),
    ('cust_2', DATE '2020-05-15',   4),
    ('cust_1', DATE '2020-05-16',  50),
    ('cust_1', DATE '2020-05-17', 100),
    ('cust_2', DATE '2020-05-18',   6);

SELECT customer_id, start_date, end_date, bottom_price
FROM memory.default.orders
MATCH_RECOGNIZE (
  PARTITION BY customer_id
  ORDER BY order_date
  MEASURES
    A.order_date AS start_date,
    C.order_date AS end_date,
    C.price      AS bottom_price
  PATTERN (A B C)
  DEFINE
    B AS B.price < A.price,
    C AS C.price < B.price
);
\end{lstlisting}

\begin{lstlisting}[language=Python,
  basicstyle=\ttfamily\small,
  backgroundcolor=\color{gray!10},
  caption={[Proposed MATCH\_RECOGNIZE on orders]Proposed engine: the same query as a \texttt{MATCH\_\allowbreak{}RECOGNIZE} string over an \texttt{orders} DataFrame.},
  label={lst:orders-mr}]
import pandas as pd
from src.executor.match_recognize import match_recognize

orders = pd.DataFrame({
    "customer_id": ["cust_1","cust_1","cust_2","cust_1",
                    "cust_2","cust_1","cust_1","cust_2"],
    "order_date":  ["2020-05-11","2020-05-12","2020-05-13","2020-05-14",
                    "2020-05-15","2020-05-16","2020-05-17","2020-05-18"],
    "price":       [100, 200, 8, 100, 4, 50, 100, 6],
})

query = """
SELECT customer_id, start_date, end_date, bottom_price
FROM orders
MATCH_RECOGNIZE (
  PARTITION BY customer_id
  ORDER BY order_date
  MEASURES
    A.order_date AS start_date,
    C.order_date AS end_date,
    C.price      AS bottom_price
  PATTERN (A B C)
  DEFINE
    B AS B.price < A.price,
    C AS C.price < B.price
); """
result = match_recognize(query, orders)
print(result)
\end{lstlisting}

\begin{lstlisting}[language=Python,
  basicstyle=\ttfamily\small,
  backgroundcolor=\color{gray!10},
  caption={[Equivalent dip detection in pandas]Python UDF: the same `dip' detection as hand-written Python/pandas.},
  label={lst:orders-udf}]
import pandas as pd

def detect_dip(df, cust_col="customer_id",
               date_col="order_date", price_col="price"):
    g = df.copy()
    g[date_col] = pd.to_datetime(g[date_col])
    g = g.sort_values([cust_col, date_col]).reset_index(drop=True)
    rows = []
    for cust, grp in g.groupby(cust_col, sort=False):
        grp = grp.reset_index(drop=True)
        i = 0
        while i + 2 < len(grp):
            a, b, c = (grp.loc[i, price_col],
                       grp.loc[i + 1, price_col],
                       grp.loc[i + 2, price_col])
            if b < a and c < b:            # A B C: successively lower prices
                rows.append({
                    "customer_id":  cust,
                    "start_date":   grp.loc[i,     date_col].date(),
                    "end_date":     grp.loc[i + 2, date_col].date(),
                    "bottom_price": c,
                })
                i += 3                     # AFTER MATCH SKIP PAST LAST ROW
            else:
                i += 1
    return pd.DataFrame(rows)

result = detect_dip(orders)
print(result)
\end{lstlisting}

\texttt{DEFINE}, and \texttt{MEASURES} clauses and covers the
simple-sequence pattern type.

\begin{lstlisting}[language=Python,
  basicstyle=\ttfamily\small,
  backgroundcolor=\color{gray!10},
  caption={[Equivalent logic in pandas]Python UDF: the same increasing-window detection as hand-written Python/pandas.},
  label={lst:stock-pandas}]
import pandas as pd

def detect_two_consecutive_increases(
    df: pd.DataFrame,
    company_col: str = "Company",
    date_col:    str = "Price_date",
    price_col:   str = "Price",
) -> pd.DataFrame:
    gdf = df.copy()
    gdf[date_col] = pd.to_datetime(gdf[date_col])
    gdf = gdf.sort_values([company_col, date_col]).reset_index(drop=True)
    out_rows = []
    for company, g in gdf.groupby(company_col, sort=False):
        g = g.reset_index(drop=True)
        inc = g[price_col].diff() > 0
        start_mask = (inc.shift(-1).fillna(False)
                      & inc.shift(-2).fillna(False))
        candidates = [int(i) for i, ok in enumerate(start_mask)
                      if ok and i + 2 < len(g)]
        match_id, next_start_pos = 0, 0
        for i in candidates:
            if i < next_start_pos:
                continue
            match_id += 1
            out_rows.append({
                "company":     company,
                "start_price": g.loc[i, price_col],
                "end_price":   g.loc[i + 2, price_col],
                "match_id":    match_id,
            })
            next_start_pos = i + 3   # AFTER MATCH SKIP PAST LAST ROW
    return pd.DataFrame(out_rows)

result = detect_two_consecutive_increases(df)
print(result)
\end{lstlisting}

\subsection{Example 3: V-Shape Recovery on the \texttt{stock\_price} Dataset}
\label{subsec:eval-vshape}
% ................................

The third example detects a \emph{V-shape recovery} on the same
table: a strictly falling run followed by a strictly rising one.
Unlike the fixed three-row patterns above, \texttt{PATTERN (A B+ C+)}
uses the \texttt{+} quantifier twice, with the fall and rise defined
through \texttt{PREV}; the measures report the start, bottom, and
recovered prices.  As before, the same detection
is expressed as a Trino SQL query (Listing~\ref{lst:vshape-trino}), as
a call to the proposed engine (Listing~\ref{lst:vshape-mr}), and as a
hand-written Python/pandas UDF (Listing~\ref{lst:vshape-udf}), and
Table~\ref{tab:vshape-results} confirms that all three, together with
Oracle, return the identical matches.

\begin{lstlisting}[basicstyle=\ttfamily\small,
  backgroundcolor=\color{gray!10},
  caption={[Trino SQL for the V-shape pattern]Trino SQL: detect the V-shape recovery on the \texttt{stock\_price} table (already created and loaded in Listing~\ref{lst:stock-trino}).},
  label={lst:vshape-trino}]
SELECT *
FROM memory.default.stock_price
MATCH_RECOGNIZE (
  PARTITION BY Company
  ORDER BY Price_date
  MEASURES
    FIRST(A.Price) AS start_price,
    LAST(B.Price)  AS bottom_price,
    LAST(C.Price)  AS end_price,
    MATCH_NUMBER() AS match_id
  ONE ROW PER MATCH
  PATTERN (A B+ C+)
  DEFINE
    B AS B.Price < PREV(B.Price),
    C AS C.Price > PREV(C.Price)
);
\end{lstlisting}

\begin{lstlisting}[language=Python,
  basicstyle=\ttfamily\small,
  backgroundcolor=\color{gray!10},
  caption={[Proposed MATCH\_RECOGNIZE V-shape]Proposed engine: the same V-shape query over the \texttt{stock\_price} DataFrame \texttt{df} of Listing~\ref{lst:stock-mr}.},
  label={lst:vshape-mr}]
from src.executor.match_recognize import match_recognize

query = """

import pandas as pd

def detect_v_shapes(df, key="Company",
                    ts="Price_date", val="Price"):
    g = df.copy()
    g[ts] = pd.to_datetime(g[ts])
    g = g.sort_values([key, ts]).reset_index(drop=True)
    out = []
    for company, grp in g.groupby(key, sort=False):
        v = grp[val].to_numpy()
        n, i, match_id = len(v), 0, 0
        while i < n - 2:
            j = i
            while j + 1 < n and v[j + 1] < v[j]:   # B+: strictly falling
                j += 1
            if j == i:                              # need at least one fall
                i += 1
                continue
            k = j
            while k + 1 < n and v[k + 1] > v[k]:    # C+: strictly rising
                k += 1
            if k == j:                              # need at least one rise
                i += 1
                continue
            match_id += 1
            out.append({
                "company":      company,
                "start_price":  int(v[i]),
                "bottom_price": int(v[j]),
                "end_price":    int(v[k]),
                "match_id":     match_id,
            })
            i = k + 1                               # AFTER MATCH SKIP PAST LAST ROW
    return pd.DataFrame(out)

result = detect_v_shapes(df)
print(result)

\paragraph{Theoretical analysis.}
Costs are stated in asymptotic notation: $O(f(n))$ is an upper bound,
$\Omega(f(n))$ a lower bound, and $\Theta(f(n))$ both at once.
Five symbols recur: $n$ input rows, $m$ pattern symbols, $d$
\texttt{DEFINE} variables, $k$ matches, $\ell$ average rows per
match.

Running a query has five stages: (1) parse the query and extract the
\texttt{MATCH\_\allowbreak{}RECOGNIZE} clauses; (2) build the
pattern's automaton (NFA, then DFA); (3) order the rows; (4) check the
\texttt{DEFINE} conditions on every row; and (5) scan for matches and
build the output.

\emph{Stages~1--2 (parse, build automaton)} depend on the pattern,
not the data: NFA construction is $O(m)$, and the NFA-to-DFA
conversion is $O(2^{m})$ states in the worst case but a handful in
practice.  For a fixed pattern this is a one-time constant.

\emph{Stage~4 (check conditions)} reads each of the $n$ rows a fixed
number of times, costing $O(nd)$.  \emph{Stage~5 (scan and output)}
costs $O(n + k\ell)$: one pass over the rows plus the work to
assemble the $k$ matches of average length $\ell$.  \emph{Stage~3
(ordering)} is the only stage that can grow faster than linearly: a
full sort costs $O(n \log n)$, but the engine sorts only when
needed---it first checks in $O(n)$ whether the rows are already
ordered and, if so, skips the sort.

Combining the data-dependent stages, the cost has \emph{two
independent axes}, and it is important to keep them apart: growth with
the data size~$n$ (for a fixed pattern), and growth with the pattern
complexity (for fixed data).

\emph{Axis~1---growth with the input size~$n$, for a fixed row-local
pattern.}  This is the axis the benchmark scales.  The automaton is
built once, and every data-dependent stage except ordering is linear,
so only whether a sort is needed separates the two cases:
\begin{itemize}
  \item \textbf{Best case} (input already ordered, as in the
    benchmark): the $O(n)$ check skips the sort, giving
    $O(n + k\ell) = \Theta(n)$.  This is optimal, because reading
    every row once is unavoidable---it meets the $\Omega(n)$ lower
    bound.
  \item \textbf{Worst case} (input unordered, a real sort required):
    $O(n \log n + nd + k\ell)$, dominated by the $O(n \log n)$ sort.
\end{itemize}
These two bracket the engine's realistic scaling and are measured
directly by the ordered and shuffled scenarios of
Section~\ref{subsec:eval-ordering}.  Memory use is $O(n + k\ell)$ in
both cases---the input, the per-row marks, and the output each grow in
proportion to the data.

Figure~\ref{fig:complexity_bigO} draws the two cases.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.72\textwidth]{images/complexity_bigO.png}
  \caption{Best-case and worst-case growth of the proposed engine.}
  \label{fig:complexity_bigO}
\end{figure}

The best case is linear and meets the $\Omega(n)$ lower bound---reading
every row once is unavoidable, so it is optimal---and holds for
already-ordered input.  The worst case is $O(n\log n)$, when a full
sort is required.  The engine's running time falls in the shaded region
between the two, depending only on whether the input arrives ordered.

\emph{Axis~2---growth with pattern complexity, for fixed data.}  A few
constructs are expensive in the pattern size~$m$ rather than in the
number of rows~$n$, and each is either paid once or held in check by an
explicit guard, so none of them appears as a growth curve in the data
size.  The NFA-to-DFA conversion is $O(2^{m})$ states in the worst
case, but it is a one-time, cached compile step capped at a fixed state
limit.  A \texttt{PERMUTE} of $m$ variables expands to $O(m!)$
branches, rejected above eight variables (or an estimated fifty
thousand branches).  And a \texttt{DEFINE} predicate that depends on
the match built so far---a running aggregate or \texttt{CLASSIFIER()},
or overlapping skip modes such as \texttt{SKIP TO NEXT ROW}---cannot be
vectorized; the engine falls back to an exact preference-ordered search
whose worst case is exponential but is bounded by a fixed
$200\,000$-step budget per match
(Section~\ref{subsec:meth-exec-backtracking}) and driven by an
explicit-stack search that does not depend on Python's recursion limit,
so it terminates over the full size range---though far more slowly than
the vectorized path (Section~\ref{subsec:eval-worst-cases}).  The
benchmark and the ordering
scenarios stay on the row-local fast path and therefore exercise
Axis~1; Section~\ref{subsec:eval-worst-cases} exercises Axis~2
empirically \emph{across all three systems}, and there the databases'
general-purpose engines overtake the proposed engine on exactly this
query class.

This section compares the engine with Trino~473 and Oracle~XE~21c on

\paragraph{Positioning.}
The engine does not aim to replace Oracle, Trino, or streaming
platforms.  It brings a practical subset of row-pattern matching into
in-memory Python and skips the load-into-a-server step.  In return it
supports only the subset of
Chapters~\ref{chap:methodology}--\ref{chap:implementation}, while the
databases have mature optimizers, distributed execution, and full SQL.
Against hand-written Pandas, though, the SQL form states partitioning,
ordering, pattern, skip, and measures in one query.

All patterns run on a derived \texttt{category} column from the product
\texttt{price}: $A$~= budget ($\le$\pounds10), $B$~= low-mid

\paragraph{Benchmark setup and resource policy.}
The benchmark is 90 cells (three systems, five families, six sizes;
1{,}800 measured executions) on the same CSVs and query bodies.

The three systems ran one at a time, with the other database containers
stopped.  Each was pinned to a single core: the engine via
\texttt{sched\_setaffinity} (with \texttt{OMP}, \texttt{OpenBLAS},
\texttt{MKL}, and \texttt{NumExpr} capped at one thread), the database
containers via \texttt{-{}-cpuset-cpus=0 -{}-cpus=1}.  Each got the same
32\,GB budget, enforced natively---an address-space \texttt{rlimit} for
the Python process, a container memory limit with no extra swap for the
databases.  The engine runs in its own cgroup, so its memory is read
from the same cgroup~v2 interface that meters the containers, and each
container is restarted before every size for a clean start (warmups
then re-warm caches and Trino's JIT).  The 32\,GB is an
engine/container cap, below the workstation's 64\,GB, so it actually
binds.

One caveat for the database runs: the Python client that issues the
query and builds the result DataFrame is also pinned to core~0, so
client and engine time-share it.  The client's memory sits outside the
32\,GB \texttt{rlimit} and is counted in neither database's memory
metric.  Reported times are client-observed wall-clock, measured
tightly around the call and stopped only once the full result set is
materialized; every system returns the same normalized rows
(Table~\ref{tab:match_counts}), so retrieval is timed on equal terms.
Database load time is excluded---the times are query execution after
loading.

The engine's throughput is stable across the range (a scan-dominated
path) and it records the lowest time on all 30 cells.  Memory uses the
two unified metrics of Section~\ref{subsec:eval-memory}: incremental
peak (per-query working memory) and absolute peak (footprint).

engine process---sampled every 50\,ms throughout the run.  Each
measured run is preceded by a five-second idle window whose average
defines the run's \emph{baseline}; the \emph{absolute peak} is the
maximum sample observed during query execution; and the
\emph{incremental peak}---the primary per-query metric---is their
difference, which removes each system's fixed resident overhead (the
already-loaded input DataFrame for the engine, Oracle's SGA, Trino's
JVM heap) from the comparison.  Like the timing figures, every
memory statistic is the mean over the twenty measured runs after
per-metric $1.5\times$IQR outlier filtering.  \emph{Query memory}
(Figure~\ref{fig:query_memory}; Table~\ref{tab:memory_cache} in
Appendix~\ref{app:benchmark-tables}) reports the incremental peak;
\emph{footprint memory} (Figure~\ref{fig:footprint_memory};
Table~\ref{tab:memory_footprint}) reports the absolute peak, which
documents the practical deployment cost of the resident engine or
container and is \emph{not} a per-query or algorithm-level metric.
Two supporting values are recorded per cell in the released
artifacts: a \emph{memory-per-million-rows} normalization (the
incremental peak divided by the input size in millions, meaningful
mainly at the larger sizes) and each database's own per-query
counter (Trino's \texttt{peakMemoryBytes}, Oracle's session-PGA
high-water delta), which serve as cross-checks below.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=\textwidth]{images/viz_query_memory.png}
  \caption[Query memory by pattern, size, and system]{Query memory
    (incremental peak) by pattern, size, and system.}
  \label{fig:query_memory}
\end{figure}

The engine's and Oracle's peaks grow linearly with size---the engine
5.7\,MB to 290\,MB at 2.22\,M rows, Oracle 2.4 to 58.7\,MB, with little
pattern dependence.  The engine's per-query memory is the largest of
the three (mean 89.7\,MB versus Oracle's 23.6\,MB, about $3.8\times$):
the cost of materializing whole-column intermediates on the vectorized
path.  It buys speed with memory.

One caveat for Trino: the JVM serves queries from its pre-committed
heap, so most of its per-query memory never crosses the OS boundary the
instrument watches.  Its reading sits at a 0.5--3\,MB floor and
\emph{understates} the true cost; its own \texttt{peakMemoryBytes}
(1.4 to 41.5\,MB) is the fairer reference.  Oracle agrees with its PGA
counter.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=\textwidth]{images/viz_footprint_memory.png}
  \caption[Footprint memory by pattern, size, and system]{Footprint
    memory (absolute peak) by pattern, size, and system.}
  \label{fig:footprint_memory}
\end{figure}

These are deployment footprints, whose boundaries differ by
construction: the engine is one Python process peaking at 0.57\,GB,
while the database values cover whole containers---Trino
8.2--14.6\,GB (the table lives in its JVM heap), Oracle near 2.0\,GB
at every size.  Per-size restarts make these independent per-size
figures.  They show deployment cost; per-query comparisons use
Table~\ref{tab:memory_cache}.

reaches about 14.6\,GB under its memory-connector setup.
Across the twenty measured runs per cell, run-to-run variability is
low: the median \emph{coefficient of variation} (the standard
deviation as a fraction of the mean) is about 1.6\% for the Pandas
engine, 3.8\% for Trino, and 2.3\% for Oracle; the five warmup runs
and the per-metric IQR rule remove the cold-start outliers that a
single warmup had left behind for the JVM-based Trino.  These
results are not presented as a claim that one system is universally
better; they show the trade-off observed under the selected
workload, data sizes, and resource configuration.

\paragraph{Why the proposed engine is faster on this workload.}
The advantage comes from specialization, not a superior execution
model.  Four factors explain it.  \emph{First}, the engine is a
single-purpose matcher: none of the planning, catalog, transaction,
or coordination machinery the databases run on every query.
\emph{Second}, execution is fully in-process over
\texttt{pandas}/NumPy arrays: no client protocol, no serialization,
no data movement.  \emph{Third}, each query compiles to a narrow
vectorized fast path (Chapter~\ref{chap:implementation}): predicates
become column operations, category checks become integer-code
comparisons, the automaton walks precompiled decision tables that
skip whole runs, and measures assemble with grouped NumPy reductions
---per-row Python work becomes amortized array work.
\emph{Fourth}, the ordered input lets the engine skip sorting;
Section~\ref{subsec:eval-ordering} shows the ranking survives
without that shortcut.  The speedup holds only for the supported
subset and single-node in-memory setting, and it costs more
per-query memory than either database
(Figure~\ref{fig:query_memory}).

Figure~\ref{fig:relative_comparison} puts the two databases against the
engine, drawn as a $1\times$ reference.

\subsection{Worst-Case Analysis}
\label{subsec:eval-ordering}
\label{subsec:eval-worst-cases}
% ................................................................

Everything so far is the engine's best case.  It has two worst cases:
when it must sort the input, and when the pattern cannot be vectorized.
This section measures both, across all three systems.

\paragraph{Worst case 1: forcing a sort.}
The benchmark orders by an already-monotonic \texttt{seq\_id}, so the
engine's $O(n)$ check skips the sort---a shortcut the databases never
take, since \texttt{ORDER~BY} always sorts.  To measure the
$O(n\log n)$ branch, the whole matrix was rerun with every input
\emph{shuffled} (deterministically), forcing a genuine sort everywhere.
The logical order is unchanged, so matches are identical across
scenarios (verified for all 90 cells) and any time difference is the
sort alone: Scenario~A (ordered) is the best case, Scenario~B
(shuffled) the fair worst case.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.78\textwidth]{images/viz_ordering_scenarios.png}
  \caption[Ordered vs shuffled input by system]{Average execution time
    on ordered vs shuffled input, by system.}
  \label{fig:ordering_scenarios}
\end{figure}

Each pair is a system's ordered (A) and shuffled (B) average; the
number above is the B/A slowdown.  Three things stand out.  The engine
takes the largest hit ($1.30\times$) because it is the only system that
skipped sorting before---widest where matching is cheap
(\texttt{simple\_sequence} $1.52\times$, \texttt{quantified}
$1.53\times$), narrowest on \texttt{complex\_nested} ($1.12\times$),
where the automaton walk dominates.  The databases slow far less
($1.13$--$1.17\times$) because they sort in \emph{both} scenarios.
Most important, the ranking holds: at 2.22\,M rows the engine is still
about $4.5\times$ faster than Oracle and $8.9\times$ than Trino under
shuffled input.  So the advantage is not an artifact of the ordered
shortcut---it survives when everyone must sort
(Tables~\ref{tab:ordering_scenarios}--\ref{tab:ordering_by_size}).

\begin{figure}[htbp]
  \centering
  \includegraphics[width=\textwidth]{images/viz_sort_cost_scaling.png}
  \caption[Ordering-step cost of the proposed engine]{Engine time on
    ordered vs shuffled input, across sizes.}
  \label{fig:sort_cost_scaling}
\end{figure}

Figure~\ref{fig:sort_cost_scaling} isolates the step: the two lines are
the engine on ordered (no sort) and shuffled (forced sort) input.  Both
grow linearly, and the shaded gap---the ordering cost---stays a small,
roughly constant slice (0.49\,s versus 0.65\,s at 2.22\,M rows), so
sorting never comes to dominate (Table~\ref{tab:sort_cost_by_size}).

\paragraph{Worst case 2: a state-dependent predicate.}
Sorting keeps the engine on its row-local fast path, where it still
wins.  The real limit is a pattern it cannot vectorize.  To show it,
the \texttt{simple\_sequence} query's \texttt{DEFINE} becomes
\texttt{B AS category='B' AND price > AVG(A.price)}---a running
aggregate.  The engine drops onto its exact backtracking searcher
(Section~\ref{subsec:meth-exec-backtracking}), while Trino and Oracle
evaluate it natively.  All three produce identical output (verified),
so the comparison is fair.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.78\textwidth]{images/viz_worstcase_xsys.png}
  \caption[Pattern-driven worst case, all systems]{Execution time on a
    state-dependent \texttt{DEFINE} query, all systems.}
  \label{fig:worstcase_coverage}
\end{figure}

Here the ranking \emph{reverses}: the engine, $2.8$--$14.5\times$
faster on the row-local benchmark, is now the \emph{slowest} at every
size---Oracle about $23$--$41\times$ faster, Trino about
$2$--$7\times$.  The cause is the specialization argument from the
other side: the engine's speed comes from compiling row-local
predicates to array operations, and \texttt{AVG(A.price)} cannot be
written that way, so it falls back to per-row search while the
databases run it natively.  For scale, the same-shaped fast-path query
takes $0.02$\,s at 100\,K rows; this one takes $1.66$\,s---about
$80\times$ slower on its own worst case.  An explicit depth-first stack
(not recursion) with a fixed exploration budget still lets the engine
finish the full range---about $64$\,s at $2.22$\,M rows versus Oracle's
$2.3$\,s and Trino's $8.7$\,s, with identical output
(Table~\ref{tab:worstcase_coverage}).

So the picture is complete: the engine is optimal $\Theta(n)$ and
fastest on the ordered row-local workload, still fastest through the
$O(n\log n)$ sorting worst case, and clearly slowest on the
pattern-driven worst case---where a database is the better tool.

\subsection{Scalability}
\label{subsec:eval-stress}
% ................................................................

Everything so far lives inside the range all three systems handle
comfortably; this section asks how far each scales.  The input is the
real Amazon Reviews~2023 corpus, taken as nested prefixes of 5, 10, 20,
40, 80, 160, and 227.9~million rows---about $100\times$ the main
dataset---with the five pattern families re-run on each.  For fairness
at scale, all three share one budget (one CPU core, 58\,GB) and each
reads the data in its own scalable mode: the engine in RAM, Trino from a
disk-backed Hive/Parquet table, and Oracle~21c Enterprise Edition from a
native heap loaded by direct path.  A query over the 600\,s per-query
budget is recorded as a wall; every completed database cell matched the
engine's output.  Figures~\ref{fig:stress_volume}
and~\ref{fig:stress_patterns} carry the results; full values are in
Tables~\ref{tab:stress_volume}--\ref{tab:stress_mem}
(Appendix~\ref{app:benchmark-tables}).

\paragraph{Time and throughput.}
All three systems scale linearly in $n$.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=\textwidth]{images/viz_stress_volume.png}
  \caption[Volume scaling on real data to 227.9M rows]{Volume scaling
    on real data: time (left) and peak memory (right).}
  \label{fig:stress_volume}
\end{figure}

Left is execution time, right peak resident memory, each the mean over
the five families.  The engine is fastest everywhere---$6$ to
$11\times$ ahead---and finishes the full corpus on every pattern
(43.6\,s for \texttt{simple\_sequence}, 124.5\,s for
\texttt{complex\_nested}); Oracle~EE finishes every cell too, while
Trino's \texttt{complex\_nested} crosses the 600\,s budget from 160\,M
rows.  On memory the roles swap: the engine holds data in RAM and is
dearest at the top (52.8\,GB), Oracle~EE stays leanest
($\approx$27\,GB), and Trino sits behind a high JVM floor
(Tables~\ref{tab:stress_volume}--\ref{tab:stress_mem}).

Figure~\ref{fig:stress_throughput} plots throughput, the mean over
families.  The engine sustains
3.7--5.2\,M\,rows/s across the whole range, against 0.5--2.8\,M for
Oracle~EE and 0.3--1.5\,M for Trino; the databases drift down as their
fixed budgets spread over more data.  The per-pattern curves stay
linear for all three (Figure~\ref{fig:stress_time_grid}), and every
completed cell returned identical match counts
(Tables~\ref{tab:stress_thr}, \ref{tab:stress_matches}).

\paragraph{Ranking.}
At the full corpus the order is engine, then Oracle~EE, then
Trino---but not uniformly.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.92\textwidth]{images/viz_stress_patterns.png}

The engine leads on all five patterns.  Oracle~EE beats Trino on the
three heavy families (\texttt{quantified} 372\,s versus 554\,s) and is
the only database to finish \texttt{complex\_nested} (498\,s); Trino
wins only on the cheapest family (\texttt{alternation}, 154\,s versus
183\,s) and time-walls on \texttt{complex\_nested} (marked).  So the
engine's vectorized matcher dominates, Oracle~EE is the more robust
database, and Trino is first to hit a time wall

\paragraph{Memory.}
The systems trade speed for memory in opposite directions.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=\textwidth]{images/viz_stress_memory.png}
  \caption[Per-query working memory on real data]{Per-query working
    memory: engine versus the databases' own counters.}
  \label{fig:stress_memory}
\end{figure}

Figure~\ref{fig:stress_memory} shows per-query working memory---the
engine's cgroup incremental peak against the databases' own counters
(Oracle's PGA, Trino's \texttt{peakMemoryBytes}).  The engine towers, its mean over families
rising to about 23\,GB (33.8\,GB on the heaviest) at 227.9\,M rows
because it keeps the working set resident, while both databases stay
under 4\,GB (Oracle near 0.5\,GB, Trino up to 3.8\,GB) as they stream
from disk.  The databases' large \emph{total} footprint in
Figure~\ref{fig:stress_volume} is fixed SGA/JVM baseline, not query
cost---so the engine's memory is the price of residency, not matcher
overhead (Tables~\ref{tab:stress_incmem}--\ref{tab:stress_native}).

\paragraph{Why the engine is scalable.}
Its time and memory are both linear in $n$ with no internal ceiling.
The only limit is machine RAM, reached predictably near the 58\,GB
budget, so the engine degrades gracefully rather than failing---and
stays $6$ to $11\times$ faster than either database on one core all the
way to 227.9\,M rows.  Oracle~EE completes the full range and is bounded
only by time on the hardest patterns; Trino hits a practical time wall
on \texttt{complex\_nested} from 160\,M.  None of this claims universal
superiority: it maps where each system's usable region lies, and the
engine wins decisively on speed at one core at the cost of being first
to exhaust memory.

\subsection{Evaluation Validity}
\label{subsec:eval-validity}
% ................................................................

A cross-system comparison only means something if the advantage cannot
be an artifact of the measurement.  Six guards make it trustworthy.

\emph{Correctness first.}  Times are compared only where the systems
return the same answer; every cell was checked for row-level equality
(Table~\ref{tab:cross_system_correctness}), so the 30/30 agreement is a
precondition, not a result.

\emph{The timer favours the databases.}  Trino is timed JIT-warm and
database load time is excluded, so the margin survives handicaps that
help the databases.

\emph{Stable over twenty runs.}  Each cell is the IQR-filtered mean of
twenty runs; the coefficient of variation is 1.6\% (engine), 2.3\%
(Oracle), and 3.8\% (Trino).  A stable mean is not a fluke.

\emph{The losing case is shown.}  We report the ordered best case, the
shuffled worst case, and the one pattern where the engine \emph{loses}
to both databases (Section~\ref{subsec:eval-worst-cases}).  Nothing is
cherry-picked.

\emph{A known cause.}  The speedup is attributed to the compiled fast
paths, found by profiling---not guessed from timings.

\emph{Narrow scope.}  The claim is only that the engine is faster
\emph{on this workload}---one partition, in-memory input, the supported
subset---on one machine.  The databases ran at default settings, so
expert tuning could narrow the gap.

\subsection{Key Findings}
\label{subsec:eval-findings}
% ................................................................

Five findings stand out.

\emph{Speed.}  The engine was fastest on all 30 cross-system cells
(0.20\,s mean versus Oracle 1.04\,s and warm Trino 1.98\,s) and stayed
$6$--$11\times$ ahead on real data to 227.9\,M rows.

\emph{Correctness.}  Every completed database cell matched the engine
row for row, so the timings compare systems computing the same answer.

\emph{Scalability.}  Its time and memory are both linear in $n$ with no
internal limit; it finished the full 227.9\,M-row corpus, bounded only
by RAM.  Oracle~EE reached it too; Trino's hardest pattern ran out of
the time budget from 160\,M rows.

\emph{The trade-off.}  Speed costs memory: the engine keeps data in
RAM, so its per-query peak is the highest and grows with $n$---the
price of residency, not overhead.

\emph{Honest limits.}  The lead is not universal: on a state-dependent
pattern both databases beat the engine.  The study maps where each
system's usable region lies; it does not crown a universal winner.

\section{Chapter Summary}
\label{sec:eval-summary}
% ----------------------------------------------------------------

This chapter evaluated the implementation through correctness,
coverage, complexity, performance, and memory analysis: five pattern
families, six sizes up to the full 2{,}222{,}742 rows, 90
cross-system cells against Trino~473 and Oracle~XE~21c under one
resource budget.

All three systems returned identical result sets in every cell.  The
engine sustained roughly 5.0\,M~rows/s with near-linear scaling and
the lowest average time (0.20\,s versus 1.04\,s for Oracle and
1.98\,s for a fully-warmed Trino), at the cost of the highest
per-query incremental memory of the three, while its operational
footprint stayed far below the resident database containers.  The
next chapter summarizes contributions, limitations, and future
work.
