\chapter{Experimental Evaluation}
\label{chap:evaluation}

This chapter evaluates the proposed
\texttt{MATCH\_\allowbreak{}RECOGNIZE}
implementation across five dimensions: output correctness validated
against Trino and Oracle on selected reference queries, coverage of
the supported SQL:2016
row-pattern subset, row-pattern type coverage, performance and
scalability, and developer usability.
Section~\ref{sec:eval-methodology}
describes the evaluation methodology, including the hardware
environment and the metrics used.
Section~\ref{sec:eval-dataset} presents the datasets and test
patterns.
Section~\ref{sec:eval-correctness} validates the concept example by
example on the small datasets.
Section~\ref{sec:eval-coverage} reports feature-coverage,
pattern-type-coverage, and usability results.
Section~\ref{sec:eval-performance} presents performance results.
Section~\ref{sec:eval-comparison} compares the system with existing
tools.

\section{Evaluation Methodology}
\label{sec:eval-methodology}

This chapter evaluates the proposed
\texttt{MATCH\_\allowbreak{}RECOGNIZE} implementation through five

% ................................
\subsection{Evaluation Criteria and Metrics}
\label{subsec:eval-metrics}
% ................................

Each test case is assessed on the five complementary criteria
introduced above, defined here together with the way each is
measured.

\paragraph{Correctness.}
The result DataFrame produced by the implementation is compared
against two independent SQL references---Trino~473 and
Oracle~XE~21c---using an exact row-level equality check (same rows,
same column values, and same order where order is deterministic).
This dimension uses the two small synthetic tables---the
\texttt{orders} table (Table~\ref{tab:orders}) for the dip example
(Table~\ref{tab:sql-match-recognize-examples}) and the
\texttt{stock\_price} table (Table~\ref{tab:stock_price}) for the
stock-price case (Section~\ref{subsec:eval-stock-price})---as well as
the Amazon benchmark data, whose 40 pattern--size results are checked
against Trino and Oracle (Section~\ref{subsec:eval-cross-system}).
The hand-written Pandas baseline is not used as a correctness oracle;
it appears only in the usability dimension
(Section~\ref{subsec:eval-usability}) to contrast the procedural
Pandas code with the single declarative
\texttt{MATCH\_\allowbreak{}RECOGNIZE} statement.

\paragraph{SQL:2016 feature coverage.}
Support for the key \texttt{MATCH\_\allowbreak{}RECOGNIZE}
clauses---\texttt{PARTITION~BY}, \texttt{ORDER~BY}, \texttt{PATTERN},
\texttt{DEFINE}, \texttt{MEASURES}, \texttt{SUBSET}, the output modes
(\texttt{ONE~ROW~PER~MATCH} / \texttt{ALL~ROWS~PER~MATCH}),
after-match skip policies, and navigation
functions~\cite{iso2016,ISO2021,monierashraf2025rowmatch}---is
verified against the implemented subset.  Each clause and sub-feature
is marked as \emph{supported}, \emph{partially supported}, or
\emph{unsupported} based on whether the implementation supports the
feature and whether the exercised behavior matches the reference
tests.  The clauses are exercised by the reference examples above and
the Trino-aligned test suite, and the resulting statuses are
summarized in Table~\ref{tab:feature_coverage}.

\paragraph{Pattern-type coverage.}
Test cases are grouped by row-pattern type---simple sequences,
quantified repetitions
(\texttt{*}, \texttt{+}, \texttt{?}, \texttt{\{n,m\}}),
ascending/descending trends, recovery patterns, alternation,
exclusion blocks (\texttt{\{-\ldots-\}}), anchored patterns, and
\texttt{PERMUTE} constructs.  A pattern type is considered covered if
at least one test case exercises it and produces correct output.  The
types are drawn from the illustrative examples and the five benchmark
pattern families, and each is mapped to its test artifact in
Table~\ref{tab:pattern_type_coverage}.

\paragraph{Performance and scalability.}
These measurements use the large real-world Amazon UK Products dataset
(Section~\ref{subsec:eval-real-dataset}) across five pattern families
and eight dataset sizes.
Execution time is recorded as \emph{client-observed end-to-end query
latency} in wall-clock seconds for each pattern--size run: it
includes result transfer and client-side DataFrame construction, and
is therefore not pure engine execution time.  Every cell of the benchmark matrix is executed
once as a warmup and then three times as measured runs; the reported
time is the \emph{median} of the three measured runs, and the
minimum and maximum are recorded alongside it to document
run-to-run spread; every reported time is thus a warmed-run latency
(measured after one warmup) rather than a cold-start time.  A single
warmup reduces cold-start effects such as import, JIT, and cache
population but is not claimed to reach full steady state.
Throughput for a cell is its dataset size divided by this median
time, and \emph{average throughput} for a system is the arithmetic
mean of its 40 per-cell throughput values.  Summary tables report
averages of these medians across pattern families or dataset sizes;
all totals and averages are computed from the full-precision
measurements and only the displayed values are rounded, so a column
of rounded entries need not sum exactly to a rounded total.  The
scaling behavior is characterized both descriptively and by an
ordinary-least-squares regression of median execution time against
row count across the eight dataset sizes, whose coefficient of
determination ($R^2$) is reported per pattern family.  Three measured
repetitions per cell keep the full benchmark matrix tractable; a
larger repetition count would provide stronger variability estimates
and is noted as a methodological refinement for future work.

\paragraph{Memory.}
Two distinct memory metrics are recorded for every run, on the same
Amazon benchmark cells used for the timing measurements.
\emph{Query memory} is the peak additional memory attributable to
executing the measured query, excluding the already-loaded data and
the idle engine; it is taken from each system's own memory
accounting (process RSS peak delta for the Pandas engine, the
per-query \texttt{peakMemoryBytes} counter reported by Trino, and
the \texttt{session pga memory max} high-water delta of a fresh
session in Oracle).  These are \emph{system-native memory
indicators} of the same concept rather than one instrument: each
engine counts memory at a different level (operating-system pages,
reserved user memory, allocated work areas), so the values support
order-of-magnitude, per-system trend comparisons, not exact
cross-system equality.  \emph{Footprint memory} is the practical
deployment footprint of having the system running at all: the
absolute process RSS peak for the Pandas engine, and for Trino and
Oracle the peak container memory sampled while the measured query
ran, on containers that keep running across dataset sizes within a
pass.  Footprint memory is reported separately and is not a
per-query or algorithm-level metric.

\paragraph{Usability (developer effort).}
The number of non-blank, non-comment lines in the equivalent
idiomatic Pandas solution is recorded and compared with the
single-statement \texttt{MATCH\_\allowbreak{}RECOGNIZE} query.  This
dimension uses the \texttt{stock\_price} example of
Section~\ref{subsec:eval-stock-price}: the declarative
\texttt{MATCH\_\allowbreak{}RECOGNIZE} statement
(Listing~\ref{lst:stock-mr}) is compared line-for-line with the
hand-written Pandas baseline (Listing~\ref{lst:stock-pandas}), and the
resulting counts are reported in Table~\ref{tab:loc_comparison}.
Hand-written Pandas solutions typically require substantially more
code (e.g.\ chained \texttt{groupby}/\texttt{shift}/joins) than a
single declarative statement, so reducing lines of code is a core
goal of the row-pattern-recognition
approach~\cite{iso2016,ISO2021,melton2017sqlrpr,Python_UDF}; the
comparison provides a concrete measure of developer-effort reduction.

\subsection{Environment and Configuration}
\label{subsec:eval-environment}
% ................................

The experiments were performed on a workstation running Ubuntu
24.04.4 LTS with an Intel Core i7-11800H processor.  For the
cross-system benchmark, every system is restricted to one CPU core
and a 32\,GB memory budget; the enforcement mechanisms are detailed
in Section~\ref{subsec:eval-complexity}.

The software environment comprised:
\begin{itemize}
  \item \textbf{Baseline SQL engines:} Trino~473 and Oracle~XE~21c
    (both run as Docker containers).
  \item \textbf{Runtimes:} Java~11.0.25 LTS and Python~3.12.7.
  \item \textbf{Core Python libraries:} \texttt{pandas}~2.3.3,
    \texttt{numpy}~2.1.3,
    \texttt{antlr4-python3-\allowbreak{}runtime}~4.13.2,
    \texttt{antlr4-\allowbreak{}tools}~0.2.1,
    and \texttt{pytest}~8.3.4.
\end{itemize}

The \texttt{MATCH\_\allowbreak{}RECOGNIZE} implementation uses ANTLR4 for
parsing Trino SQL, \texttt{pandas} for DataFrame management, and
custom AST/NFA/DFA modules for pattern detection (see
Chapter~\ref{chap:methodology}).

Unless stated otherwise, all experiments use this configuration.  For
the correctness comparisons in particular, equivalent input data are
loaded into Trino, Oracle, and a Pandas DataFrame using matching
schemas, and the result sets are then compared by row order and
measure values after normalizing case-only column-name differences
between the SQL and Pandas outputs.

\paragraph{Dataset 3: the Amazon UK Products dataset.}
The Amazon UK Products Dataset is a large e-commerce dataset
containing information about products listed on Amazon's UK
marketplace~\cite{asaniczka2023amazon}, commonly used for data
analysis, machine learning, and benchmarking.  It is provided as a
single CSV file with roughly 2.2\,million product records.  Each
record includes: \texttt{asin} (Amazon Standard Identification Number;
unique product ID); \texttt{title} (product name); \texttt{imgUrl}
(primary product image URL); \texttt{productURL} (canonical product
page URL); \texttt{stars} (average customer rating, 1--5,
floating-point); \texttt{reviews} (number of customer reviews,
non-negative integer); \texttt{price} (listed price in GBP, numeric);
\texttt{isBestSeller} (bestseller indicator, boolean);
\texttt{boughtInLastMonth} (approximate purchases in the last 30 days,
integer); and \texttt{categoryName} (top-level category label, e.g.\
Electronics, Books, Clothing).  This dataset is representative for
price-trend analysis, brand/feature-popularity studies, pattern
recognition, and recommender-system benchmarking.  It supplies the
data volume for the performance and scalability benchmark
(Section~\ref{subsec:eval-real-dataset}); the derived
\texttt{category} column and the construction of the eight dataset
sizes are described with the benchmark patterns in
Section~\ref{subsec:eval-pattern-defs}.

\section{Validation of Concept}
\label{sec:eval-correctness}
% ----------------------------------------------------------------

This section validates the implementation example by example and
dataset by dataset.  Each worked example runs the same
\texttt{MATCH\_\allowbreak{}RECOGNIZE} query through the proposed engine

proposed engine, Trino, and Oracle producing identical results
throughout; the same references also serve for all 40 queries of the
cross-system benchmark in Section~\ref{subsec:eval-cross-system}.

\subsection{Example 2: Stock-Price Analysis on the \texttt{stock\_price} Dataset}
\label{subsec:eval-stock-price}
% ................................

The second example uses the \texttt{stock\_price} table (Dataset~2,
Table~\ref{tab:stock_price}) to demonstrate and validate
pattern-matching queries with the
\texttt{MATCH\_\allowbreak{}RECOGNIZE} clause.

This example detects bullish momentum by identifying a three-row
window in which the second and third rows increase relative to their
immediate predecessors.  The query partitions records by company
(\texttt{PARTITION BY company}), orders them chronologically
(\texttt{ORDER BY price\_date}), and applies \texttt{PATTERN (A B C)}
with \texttt{DEFINE} predicates on \texttt{B} and \texttt{C}.  The
variable \texttt{A} has no explicit \texttt{DEFINE} predicate and
therefore uses the standard implicit always-true behavior.  Key
attributes---starting price, ending price, and match ID---are produced
in the \texttt{MEASURES} clause.  A structured \texttt{pandas}
DataFrame serves as input to \texttt{MATCH\_\allowbreak{}RECOGNIZE}, enabling
automated detection of upward trends across firms for the supported
row-pattern subset.
Additional use cases are available in the project
repository~\cite{monierashraf2025rowmatch}.

\noindent\textbf{Load the dataset as a Pandas DataFrame.}

\noindent\textbf{Python DataFrame Baseline.}
As a Python baseline, the same pattern logic was implemented directly in
\texttt{pandas}, without using \texttt{MATCH\_\allowbreak{}RECOGNIZE}.  The input
DataFrame (Listing~\ref{lst:dataset}) is converted to \texttt{datetime}
and sorted by company and trading date.  For each company, the
baseline computes a Boolean signal
\texttt{inc\,:=\,diff(Price)\,>\,0} and marks candidate
starts where the next two differences are positive, corresponding to
the SQL pattern rows $A,B,C$ with $B>A$ and $C>B$.  Candidates are
processed sequentially with a
non-overlapping policy, advancing
\texttt{next\_start\_pos\,=\,i\,+\,3} to emulate
\texttt{AFTER MATCH SKIP PAST LAST ROW}.  For each accepted start, the
implementation outputs \texttt{company}, \texttt{match\_id}, and the
measures \texttt{start\_price} from the start row~(A) and
\texttt{end\_price} from the final row of the match~(C), returning
the resulting matches as a \texttt{DataFrame}~\cite{Python_UDF}.

\begin{lstlisting}[language=Python,
  basicstyle=\ttfamily\small,
  backgroundcolor=\color{gray!10},
  caption={[Equivalent logic in pandas]Equivalent pattern matching in Python/pandas.},
  label={lst:stock-pandas}]
import pandas as pd

def detect_two_consecutive_increases(
    df: pd.DataFrame,
    company_col: str = "Company",
    date_col:    str = "Price_date",
    price_col:   str = "Price",
) -> pd.DataFrame:
    gdf = df.copy()
    gdf[date_col] = pd.to_datetime(gdf[date_col])
    gdf = gdf.sort_values([company_col, date_col]).reset_index(drop=True)
    out_rows = []
    for company, g in gdf.groupby(company_col, sort=False):
        g = g.reset_index(drop=True)
        inc = g[price_col].diff() > 0
        start_mask = (inc.shift(-1).fillna(False)
                      & inc.shift(-2).fillna(False))
        candidates = [int(i) for i, ok in enumerate(start_mask)
                      if ok and i + 2 < len(g)]
        match_id, next_start_pos = 0, 0
        for i in candidates:
            if i < next_start_pos:
                continue
            match_id += 1
            out_rows.append({
                "company":     company,
                "start_price": g.loc[i, price_col],
                "end_price":   g.loc[i + 2, price_col],
                "match_id":    match_id,
            })
            next_start_pos = i + 3   # AFTER MATCH SKIP PAST LAST ROW
    return pd.DataFrame(out_rows)

result = detect_two_consecutive_increases(df)
print(result)
\end{lstlisting}

\section{Comparison with Existing Systems}
\label{sec:eval-comparison}
% ----------------------------------------------------------------

The evaluated system is not intended to replace full SQL engines such
as Oracle or Trino, nor streaming platforms such as Flink.  Its design
goal is narrower: it brings a practical subset of SQL row-pattern
matching into in-memory Python analytics workflows over
\texttt{pandas} DataFrames.

Compared with Trino and Oracle, the proposed implementation avoids the
need to load data into a database server before running
\texttt{MATCH\_\allowbreak{}RECOGNIZE}.  This makes it convenient for notebook and
DataFrame-centered analysis, but it also means that the implementation
supports only the subset described in Chapters~\ref{chap:methodology}
and~\ref{chap:implementation}.  Trino and Oracle remain more complete
SQL systems with mature optimizers, distributed execution, and broader
standard coverage.  Compared with hand-written Pandas logic, the SQL
form is more declarative and reusable: partitioning, ordering, pattern
variables, skip behavior, and measures are expressed in one query
rather than in custom procedural code for each pattern.

\paragraph{Theoretical analysis.}
As implemented, the execution path has five main steps:

\begin{enumerate}
  \item parse the SQL query and extract the
    \texttt{MATCH\_\allowbreak{}RECOGNIZE} clauses;
  \item compile the row pattern into an internal automaton
    (NFA, then DFA);
  \item partition the input and order each partition;
  \item vectorize the \texttt{DEFINE} predicates over the ordered
    rows and derive the set of positions that can start a match;
  \item scan the candidate start positions, run the automaton, and
    build the output matches and measures.
\end{enumerate}

Let $n$ be the number of input rows, $p$ the number of partitions,
$n_i$ the number of rows in partition $i$, $d$ the number of
\texttt{DEFINE} variables, $k$ the number of matches produced, and
$\ell$ the average number of rows returned per match.  Thus,
$\sum_{i=1}^{p} n_i = n$.  For the five benchmark patterns used in
this chapter the pattern is fixed while the dataset size grows from
50\,K to 2\,M rows.

Steps~1 and~2 depend only on the query, not on $n$.  Automaton
compilation exhibits combinatorial worst-case growth in the pattern
size---exponential in general, and factorial for \texttt{PERMUTE},
which enumerates all permutations of its arguments---so the
implementation applies complexity guards and conservative fallbacks;
for the fixed benchmark patterns this cost is a small constant paid
once.

Step~3 is the ordering step, and the implementation treats it
conditionally rather than always sorting.  Each partition is first
tested for monotonicity in the \texttt{ORDER~BY} columns
($O(n_i)$); \texttt{pandas} \texttt{sort\_values} is invoked
\emph{only} when the partition is not already ordered, at a cost of
\[
  O\!\left(\sum_{i=1}^{p} n_i \log n_i\right) \le O(n \log n).
\]
When every partition is already ordered the sort is skipped and the
step reduces to the $O(n)$ monotonicity check.  The benchmark queries
order by a sequential \texttt{seq\_id} whose values are already
monotonic, so this run takes the $O(n)$ path---which is precisely why
the measured scaling is approximately linear rather than
$n \log n$.

Step~4 evaluates the row-local \texttt{DEFINE} predicates as
vectorized column operations, producing a Boolean start-position mask
in $O(nd)$ time and letting the scan skip positions that cannot begin
a match; $d$ is a small constant for the benchmark patterns.

Step~5 visits the candidate start positions.  Each match attempt runs
the automaton over the rows it consumes, using $O(1)$ precomputed
predicate lookups and a greedy plan for consecutive quantifiers
(e.g.\ \texttt{A+ B+}) that avoids exponential backtracking.  Under
the default \texttt{AFTER MATCH SKIP PAST LAST ROW} used by the
benchmark, accepted matches are non-overlapping and the scan advances
past each match, so the total scan and output work is
\[
  O(n + k\ell),
\]
where $O(n)$ covers the position scan and $O(k\ell)$ the construction
of matched rows and measures.  Combining the steps, the conservative
end-to-end bound is
\[
  O(n \log n + nd + k\ell),
\]
which on already-ordered input such as the benchmark data---where the
sort is skipped and the ordering step is the $O(n)$ monotonicity
check---reduces to $O(n + nd + k\ell)$, and collapses further to
$O(n + k\ell)$ (linear in $n$) for fixed $d$.  This
bound is not universal: overlapping skip modes (e.g.\ \texttt{SKIP TO
NEXT ROW}) and heavily ambiguous patterns can revisit rows and raise
the per-position cost above the linear regime.

For daily workloads, where new rows often arrive out of order, the
most practical future improvement is incremental ordering.  Historical
data can be stored in sorted form; when a new daily batch arrives,
only the new batch is sorted and then merged with the already ordered
history.  This does not remove ordering completely, but it avoids
sorting the full history again for every run.

The extra space used by the matcher comes mainly from the partition
and ordering structures, the compiled automaton and vectorized
predicate matrix, and the output rows:
\[
  \text{extra space} \approx O(n + \text{pattern state} + k\ell).
\]
The input DataFrame itself also uses memory proportional to the number
of rows and columns.  Because the benchmark uses no
\texttt{PARTITION~BY} clause, it forms a single partition and runs on
one core; the engine's parallel-partition path is engaged only when
more than one partition is present and is therefore not exercised
here.


\paragraph{Empirical and cross-system analysis.}
The empirical analysis measures the implemented systems directly.  In
this chapter, the final benchmark consists of 120 successful
benchmark cells: three systems, five pattern families, and eight
dataset sizes from 50\,K to 2\,M rows, where each cell is the median
of three measured executions after one warmup (360 measured
executions in total).  The proposed Pandas engine, Trino~473, and
Oracle~XE~21c were evaluated on the same generated CSV datasets and
the same \texttt{MATCH\_\allowbreak{}RECOGNIZE} query bodies.

The resource policy was enforced as follows.  The three systems ran
strictly sequentially, one at a time, with the other database
containers stopped.  The Pandas engine was pinned to CPU core~0 via
\texttt{sched\_setaffinity}, with the numerical-library
thread pools (\texttt{OMP}, \texttt{OpenBLAS}, \texttt{MKL},
\texttt{NumExpr}) limited to one thread; the database containers
were pinned to the same core with
\texttt{docker update -{}-cpuset-cpus=0} combined with a one-CPU
quota (\texttt{-{}-cpus=1}).  Every system was given the same
nominal 32\,GB memory budget, enforced by the mechanism native to
each deployment: an address-space \texttt{rlimit} for the Python
process, and a container memory limit with \texttt{-{}-memory-swap}
equal to the limit (no extra swap) for the databases.  These
mechanisms are equivalent in budget but not identical in enforcement
semantics.  The 32\,GB figure is therefore an
\emph{engine/container} memory budget: it caps the Pandas engine
process and each database container.  The budget is below the
workstation's 64\,GB of installed RAM, so it is an actually enforced
cap.  For the database runs, the separate Python client that issues
the query and materializes the result DataFrame is pinned to core~0
via \texttt{sched\_setaffinity} (the same call that pins the Pandas
engine), applied once to the benchmark process at start-up; container
pinning alone would not constrain this host-side client, and the
client's own memory is not itself placed under the 32\,GB
\texttt{rlimit}.  Client and database engine therefore time-share the
single core~0, and, correspondingly, the container footprint values
cover only the database process and exclude this client, whose
result-construction memory is counted in neither database memory
metric.
Reported times are client-observed wall-clock
durations of query execution including result retrieval, measured
tightly around the query call so that sampling overhead was
minimized.  The timer
stops only after the client has fully materialized the result set
(fetching all rows and constructing the client-side DataFrame), and
every system returns the same normalized result columns and row
counts for each cell (Table~\ref{tab:match_counts}), so result
retrieval is included in the timing on equal terms.  Database
loading time is excluded from the reported query times; the
measurements report query execution after the data has been loaded
into the target system.

The theoretical model explains the expected growth, while the empirical
results show the current implementation's observed behavior.  The
Pandas engine shows stable throughput across the evaluated range,
which is consistent with a scan-dominated execution path for these
fixed-size patterns, while Trino and Oracle sustain higher average
throughput on most pattern--size combinations, with
\texttt{complex\_nested} as the main exception for Trino.  Memory is reported using the two metrics defined in
Section~\ref{subsec:eval-metrics}: the engine-reported
system-native \emph{query-memory indicators}, and the
\emph{operational footprint}, which documents the cost of the
resident process or database container and is reported separately.

% ................................................................

Memory is reported using the two metrics defined in
Section~\ref{subsec:eval-metrics}.  \emph{Query memory}
(Table~\ref{tab:memory_cache}) is the peak additional memory
attributable to executing the measured query, taken from each
system's own memory accounting: process RSS peak delta for the
Pandas engine, the per-query \texttt{peakMemoryBytes} counter for
Trino, and the session PGA high-water delta of a fresh session for
Oracle.  All three exclude the stored table data and the idle
engine.  They are, however, \emph{system-native indicators} rather
than one instrument: RSS counts operating-system pages, Trino counts
reserved user memory, and Oracle counts allocated work-area memory.
The values should therefore be read as order-of-magnitude and
per-system trend evidence, not as directly comparable measurements
of identical quantities.

absolute process RSS peak for the Pandas engine and the Docker
container peak for Trino and Oracle; it documents the practical
deployment cost of the resident engine and is \emph{not} a
per-query or algorithm-level metric.

Table~\ref{tab:memory_cache} shows that the average query-memory
indicators generally increase with input size, as expected for a
per-query working set.  For the most memory-intensive pattern
families at the largest input (2\,M rows), the three indicators fall
between roughly 40 and 80\,MB; this range describes only those
worst-case cells, not the benchmark as a whole, where most readings
are far smaller.  The Pandas engine recorded the smallest
system-native indicator values on the sequence-style pattern
families, while \texttt{optional\_pattern} and
\texttt{complex\_nested}, which produce the largest match sets,
bring its readings into the same order of magnitude as the database
engines; at 2\,M rows the engine's \texttt{complex\_nested}
indicator (79.84\,MB) is about 1.5$\times$ Oracle's (53.19\,MB) and
about 2.0$\times$ Trino's (39.63\,MB).

Values of 0.00 in Table~\ref{tab:memory_cache} do not mean that a
query executed without memory; they mean that the query's working
set stayed below the resolution of the corresponding counter.
Oracle pre-allocates an initial block of PGA to every new session at
connect time, so a small query whose \texttt{MATCH\_\allowbreak{}RECOGNIZE}
work areas fit inside that initial allocation never raises the
session's PGA high-water mark, and the measured delta is zero.
Likewise, a Pandas run that fits into pages already resident in the
Python process produces no RSS growth.  Readings close to this floor
are also quantized to the runtime's allocation granularity (Oracle,
for example, extends session PGA in small fixed-size chunks), so
near-floor values should not be compared against each other.  At larger input sizes the database counters
grow steadily with the input, while the RSS-based Pandas readings
fluctuate more (values can rise, fall, or return to zero between
neighboring cells) because RSS deltas depend on page reuse and
garbage-collection timing inside the Python process; the overall
upward trend beyond the measurement floor is what the metric is
intended to show.

Table~\ref{tab:memory_footprint} shows the practical deployment
footprints.  The boundaries differ by construction: the Pandas value
covers one Python process including the loaded input DataFrame,
whereas the database values cover entire Docker containers.  The
Pandas engine stays below 1\,GB even at 2\,M rows.  Trino's
container grows to about 27\,GB: the benchmark table is held by its
memory connector inside the JVM heap, the table is dropped and
recreated before each dataset size is loaded, and the container
keeps running across all sizes within the system's benchmark pass
(it is restarted only between systems).  Each database value is the
peak container memory sampled during that measured run, not a
monotonic high-water counter: because the JVM rarely returns freed
heap to the operating system, the readings carry allocations
retained from earlier sizes, yet they can also fluctuate downward
between neighboring cells because the container-level usage metric
excludes reclaimable page cache that the kernel may free at any
time.  Oracle's container stays near 2.2--2.5\,GB.  The two kinds
of values therefore have different reset boundaries: the Pandas
readings are per-run process peaks, while the database readings are
per-run samples of a container whose memory state resets only when
the container is restarted at the start of the pass, and they must
not be interpreted as independent per-size footprints.  Restarting each container before every dataset
size would yield independent per-size footprint measurements and is
the recommended refinement for future benchmark runs; it was not
applied here to keep the pass duration tractable.  These values
document what it costs to have each system running as deployed here;
per-query comparisons should use Table~\ref{tab:memory_cache}.

Tables~\ref{tab:overall_stats}--\ref{tab:cross_system_correctness}
summarize the main findings.  All 120 benchmark cells completed
successfully, and Trino and Oracle produced output identical to the
proposed engine's normalized output for all 40 pattern--size
combinations each---all three systems produced identical result
sets.  Oracle had the lowest average
execution time in this benchmark (0.82\,s), followed by Trino
(1.91\,s) and the proposed Pandas engine (2.01\,s): the engine's
overall average is within about 5\% of Trino's, and the engine is
faster than Trino on two of the five families
(\texttt{alternation} and \texttt{complex\_nested}, the latter by
roughly a factor of two), while Oracle sustains above one million
rows per second on most combinations.  The system-native
query-memory indicators of all three systems fall in the same order
of magnitude, reaching at most 40--80\,MB for the most
memory-intensive patterns at 2\,M rows---where the engine's
\texttt{complex\_nested} indicator (79.84\,MB) is about
1.5$\times$ Oracle's and 2.0$\times$ Trino's---while the
operational
footprints differ substantially: the Pandas engine peaks below
1\,GB as a single Python process, the Oracle container stays near
2.5\,GB (about 2.7$\times$ the Pandas peak), and the Trino
container reaches about 27\,GB under its memory-connector setup.
The recorded per-cell minimum and maximum times (released with the
benchmark artifacts) show a median run-to-run spread of about 9\%
for the Pandas engine, 20\% for Trino, and 5\% for Oracle.  These
results are not presented as a claim that one system is universally
better; they show the trade-off observed under the selected
workload, data sizes, and resource configuration.

\section{Chapter Summary}
\label{sec:eval-summary}
% ----------------------------------------------------------------

This chapter evaluated the proposed \texttt{MATCH\_\allowbreak{}RECOGNIZE}
implementation through correctness checks, feature-coverage tests,
complexity analysis, performance measurements, memory analysis, and
comparison with existing systems.  The experiments used five pattern
families across eight dataset sizes, with every cell reported as the
median of three measured runs, producing 40 successful benchmark
cells for the proposed engine and a 120-cell cross-system comparison
against Trino~473 and Oracle~XE~21c under the same nominal resource
budget.

The results show that the implementation preserves the evaluated
semantics for the benchmark pattern families---Trino and Oracle
produced output identical to the engine's normalized output in all
40 pattern--size combinations each (all three systems returned
identical result sets)---while reducing
developer effort compared with hand-written Pandas logic.  Under the
selected workload and resource configuration, the proposed engine
sustained stable throughput of roughly 457\,000~rows/s with
approximately linear scaling, an average execution time within about
5\% of Trino's (Oracle was the fastest system in this benchmark),
query-memory indicator values in the same order of magnitude as the
database engines, and a substantially smaller operational footprint,
since it runs inside a single Python process without a resident
database server.  The next chapter summarizes the thesis
contributions, discusses limitations, and outlines future work.
