\chapter{Introduction}
\label{chap:intro}
Pattern matching means looking at a series of data points, like those in a table, to find shapes such as V, W, and U, as well as trends and patterns in behavior. This approach helps analysts discover useful insights for analysis and prediction. Pattern matching is used in many fields, including finding money-laundering transaction chains~\cite{petkovic2022fraud}, spotting breakdowns in IoT sensor data, and following user journeys in e-commerce clickstreams. While standard SQL clauses look at each row separately, pattern matching pays attention to the order of events and how rows relate to each other. Patterns are set up using syntax that works like regular expressions, but at the row level. This lets analysts find and pull out meaningful sequences from ordered groups of data.

\section{Problem Statement}
\label{sec:intro-problem}
% ----------------------------------------------------------------
The\texttt{MATCH\_RECOGNIZE}(MR) clause was introduced in the SQL:2016 standard, enabling users to define and search for patterns within ordered sequences of rows directly in SQL queries. Although row pattern recognition was standardized in SQL:2016~\cite{iso2016,ISO2021}, Oracle Database had already introduced support for  \texttt{MATCH\_RECOGNIZE}  (in Oracle Database~12c, 2013) ~\cite{oracle2022,book}. However, only a limited number of database management systems have since adopted this functionality.

Several modern analytical platforms have since emerged. Trino~\cite{trino2022,trino2021row},, Snowflake~\cite{snowflake2025match_recognize}, and Apache Flink ~\cite{flink} each offer their own implementations; however, conformance to the SQL:2016 feature levels (R010, R020, R030) varies with different restrictions. This limitation is also still under development in PostgreSQL and DuckDB, two widely used open-source systems that do not currently provide native support for  \texttt{MATCH\_RECOGNIZE} ~\cite{howe2020matchrecognize,findling2025bringing}.

This gap affects analytical workflows. Many analytical workflows begin in Python, where users prepare data, test hypotheses, and build models before transitioning to larger database or cloud platforms. For data that fits in memory, using a separate SQL engine to express row-pattern logic can add overhead for deployment and data movement. The focus of this thesis is therefore not to replace enterprise or Database systems, but to bring a practical subset of declarative row-pattern recognition into the Pandas environment, where many analytical workflows already begin.

If you do not have this declarative clause, finding multi-row patterns means using complicated mixes of SQL window functions, self-joins, or even looping through rows outside the database. When the pattern has more than two or three steps, the code quickly grows into many lines with row number calculations, timestamp checks, and lots of \texttt{JOIN} conditions. This kind of code is hard to write, even harder to maintain, and often leads to multiple table scans that hurt performance~\cite{cao2012optimizationanalyticwindowfunctions}.

This limitation stands out in Python’s data science tools. Pandas~\cite{McKinney2010}, a popular library for working with tables, does not have built-in support for sequential pattern matching. If analysts want to find multi-step patterns, they have to do it themselves by creating shifted copies of columns, such as \texttt{df['value'].shift(-1)}, \texttt{shift(-2)}, and so on, making Boolean masks, using \texttt{groupby().apply()} to piece together sequences, and then pivoting or renaming columns.
Listing~\ref{lst:ch1-udf} illustrates this
approach for a simple three-step decreasing-price pattern (\texttt{A~B~C}).

\begin{lstlisting}[language=Python,
    caption={Detecting a three-step price-decrease pattern in
             Pandas \emph{without} \texttt{MATCH\_RECOGNIZE}},
    label={lst:ch1-udf}]
import pandas as pd

df['order_date'] = pd.to_datetime(df['order_date'])
df = df.sort_values(['customer_id', 'order_date']).reset_index(drop=True)

def match_pattern_ABC(group: pd.DataFrame):
    matches = []
    for i in range(len(group) - 2):
        A, B, C = group.iloc[i], group.iloc[i+1], group.iloc[i+2]
        if (B['price'] < A['price']) and (C['price'] < B['price']):
            matches.append({
                'customer_id':  A['customer_id'],
                'start_date':   A['order_date'],
                'end_date':     C['order_date'],
                'bottom_price': C['price']
            })
    return matches

all_matches = []
for _, grp in df.groupby('customer_id'):
    all_matches.extend(match_pattern_ABC(grp))

result_df = pd.DataFrame(all_matches)
\end{lstlisting}

 It takes over 20 lines of code to make a simple three-step pattern. The Hand-Crafted Approach is also likely to have mistakes where things are not lined up correctly. The Hand-Crafted Approach does not work well when we need to make longer patterns or patterns that have conditions.

The same query expressed with \texttt{MATCH\_RECOGNIZE} is shown
in Listing~\ref{lst:ch1-mr}: a single declarative statement that
specifies the partition, ordering, pattern grammar, variable definitions, and output measures.

\begin{lstlisting}[language=Python,
    caption={The same pattern expressed with \texttt{MATCH\_RECOGNIZE}
             on a Pandas DataFrame},
    label={lst:ch1-mr}]
import pandas as pd
from pandas_match_recognize import match_recognize

df['order_date'] = pd.to_datetime(df['order_date'])

query = """
SELECT customer_id, start_date, end_date, bottom_price
FROM orders
MATCH_RECOGNIZE (
  PARTITION BY customer_id
  ORDER BY     order_date
  MEASURES
    A.order_date AS start_date,
    C.order_date AS end_date,
    C.price      AS bottom_price
  ONE ROW PER MATCH
  AFTER MATCH SKIP TO NEXT ROW
  PATTERN (A B C)
  DEFINE
    B AS B.price < A.price,
    C AS C.price < B.price
)
"""

output_df = match_recognize(query, df)
\end{lstlisting}

The declarative form is not just shorter in syntax: it helps to separate what you need to find, which is the pattern,from how you find it. The patterns become composable, auditable, and easy to extend. A more complex pattern, such as four or five steps
with conditional alternation, requires no change to the surrounding
Python code, only an update to the \texttt{PATTERN} and
\texttt{DEFINE} clauses~\cite{michels2018new,petkovic2022rpr}.
It also allows the implementation to compile the row pattern into
automata, cache repeated pattern structures, and process ordered
partitions without exposing this execution logic to the user.

The full structure and sub-clauses of \texttt{MATCH\_RECOGNIZE}
are introduced in Chapter~\ref{chap:background}.  Chapter~\ref{chap:related}
surveys the existing implementations and identifies the research
gap.

\section{Research Objectives and Contributions}
\label{sec:intro-objectives}
% ----------------------------------------------------------------
This thesis explores how to bring SQL:2016 \texttt{MATCH\_RECOGNIZE} row pattern recognition into Python data analysis with Pandas. The goal is to combine the strengths of relational pattern queries and flexible in-memory analytics. The work was guided by three main objectives, each matched with a specific contribution described below.

\begin{enumerate}

\item \textbf{An In-Process Engine Supporting Core SQL:2016 Feature~R010-Style Capabilities for Pandas DataFrames.}

Without a declarative alternative in Python, a data scientist must write the many lines of hand-crafted loops like in Listing~\ref{lst:ch1-udf} to detect a simple pattern like three steps declining-price pattern. This thesis delivers an in-process \texttt{MATCH\_RECOGNIZE}-style implementation supporting the core Feature~R010 behavior evaluated in this thesis. It runs entirely alongside Pandas, with no external database server or cloud subscription required, reducing that loop to a single declarative call like in Listing~\ref{lst:ch1-mr} and avoiding the requirement for external SQL engine dependencies or intricate Python code.

\item \textbf{Broad Coverage and Cross-Engine Correctness Validation.}

The engine works with many SQL:2016-style row-pattern features. It handles quantifiers (\texttt{*}, \texttt{+}, \texttt{?}, and \texttt{{n,m}}), alternation (\texttt{|}), grouping, anchors, exclusions (\texttt{{- -}}), and \texttt{PERMUTE}. For example, a fraud analyst might describe a probe-and-drain sequence like this:

\begin{lstlisting}[language=SQL]
PATTERN (PROBE+ (TRANSFER | WITHDRAW){2,5} DRAIN)
\end{lstlisting}

The engine reads, compiles, and runs the pattern directly on pandas DataFrames. We check that it works correctly using focused automated tests and by comparing results with Oracle and Trino.

More than 500 queries were run using different pattern families, SQL constructs, and output scenarios. For the cases tested, the proposed engine gave results that matched those from Oracle and Trino, including detected matches, match order, measures, and output rows.

\item \textbf{Empirical Validation Using a Real-World Dataset.}

We assess performance with targeted automated tests, using the Amazon UK Products~2023 catalog~\cite{asaniczka2023amazon}. This real-world e-commerce dataset has about 2.2 million product records and offers good tabular data for testing price-based row-pattern recognition.

We evaluated performance using five pattern families and six input sizes, from 100,000 rows up to the full 2,222,742-row Amazon UK Products catalog, constructed as nested prefixes of the source data.

A total of 30 pattern--size benchmark cells were measured for the proposed engine (90 cells across the three compared systems), and every cell completed successfully. The results show almost linear scaling across the tested range, with the expected differences in runtime for simple, alternation, quantified, optional, and nested pattern types; under the shared one-core, 32\,GB resource budget the engine recorded the lowest average execution time of the three systems, while the two database references validated its output row-for-row.

\end{enumerate}

\section{Organization of the Thesis}
\label{sec:thesis-organization}
% ----------------------------------------------------------------

The following section outlines the structure of the thesis.
\textbf{Chapter~\ref{chap:background} --- Background and Concepts}
presents the theoretical and technical foundations on which this work is built.  It introduces sequential pattern matching and its
real-world applications, explains the SQL:2016
\texttt{MATCH\_RECOGNIZE} clause and its eight sub-clauses in
detail, and introduces the SQL:2016 feature-level classification
(R010, R020, R030)
used throughout the thesis.

\textbf{Chapter~\ref{chap:related} --- Related Work}
surveys each major \texttt{MATCH\_RECOGNIZE} implementation in
depth---Oracle, Trino, Snowflake, Apache Flink, and DuckDB---documenting
their documented feature scope, execution strategies, deployment
models, and practical
restrictions.  It also examines the Python and Pandas ecosystem
gap and formally positions the contribution of this thesis within
the broader pattern-matching research landscape.

\textbf{Chapter~4 --- System Design and Methodology}
describes the overall system architecture and methodology.  It
covers the SQL parsing pipeline, grammar and AST design, semantic
validation, logical plan construction, and the design choices that
connect declarative \texttt{MATCH\_RECOGNIZE} queries to the
execution layer.

\textbf{Chapter~5 --- Implementation and Optimization}
This chapter explains how the engine is built. It looks at finite-automata compilation, how NFAs and DFAs are created, how Pandas is used to run processes, how outputs are produced, how measures are evaluated, and the performance layer: compiled per-query fast paths (vectorized condition evaluation, row-local DFA decision tables, a linear-pattern search, and a fused \texttt{ONE ROW PER MATCH} driver) together with pattern caching and memory management.


\textbf{Chapter~6 --- Experimental Evaluation}
presents the evaluation methodology, the benchmark dataset
(Amazon UK Products 2023, 2.2~million rows), and experimental
results covering functional correctness, performance across five
pattern families and six input sizes, scalability, pattern
complexity, and comparison with existing systems.

\textbf{Chapter~7 --- Conclusions and Future Work}
This chapter summarizes the main contributions, discusses the limitations of the current implementation, and suggests directions for future research and further extensions.
