\chapter{System Design and Methodology}
\label{chap:methodology}

The \texttt{MATCH\_RECOGNIZE} implementation follows a modular,
five-stage architecture specifically designed for SQL row-pattern
recognition.  The engine executes SQL queries directly on
\texttt{pandas} DataFrames~\cite{monierashraf2025rowmatch} while
preserving the supported SQL:2016 row-pattern semantics~\cite{iso2016,ISO2021,melton2017sqlrpr}.
The pipeline consists of five phases: \emph{parsing},
\emph{semantic validation}, \emph{logical plan construction},
\emph{pattern compilation}, and \emph{execution}
(Fig.~\ref{fig:system-architecture}).

In the parsing stage, the SQL query is transformed into an abstract
syntax tree (AST) that captures both the \texttt{SELECT}/\texttt{FROM}
clauses and the \texttt{MATCH\_RECOGNIZE}
sub-clauses~\cite{tarazona2021antlr,parr2013definitive}.  Semantic
validation then checks variable usage, predicates, skip policies,
output modes, and the consistency of the referenced row-pattern
constructs.  The validated AST becomes a logical plan, from which the
system compiles an NFA with Thompson's construction---extended with
\texttt{DEFINE} predicates, prioritized $\varepsilon$-transitions,
and subset construction with state
deduplication~\cite{salomaa1996nfa}---and finally executes the
compiled automaton over ordered partitions, materializing matches
with the correct \texttt{RUNNING}/\texttt{FINAL} semantics directly
as \texttt{pandas} DataFrames.

% ----------------------------------------------------------------
\section{System Architecture Overview}
\label{sec:meth-overview}
% ----------------------------------------------------------------

The engine's five-phase pipeline is designed so that each stage
produces a well-defined artifact consumed by the next, and it rests
on three primary runtime components.  The \emph{parser} analyzes the
SQL query, recognizes its components---partitioning, ordering, the
pattern, conditions, and output measures---and organizes them into
an AST that eases validation and prepares the query for matching.
The \emph{automata compiler} converts the pattern into an NFA and
then, for efficiency, into a DFA that removes transition ambiguity
and handles repetitions, alternatives, and sequences in a uniform
way~\cite{georgiou2025streaming}.  The \emph{execution engine}
partitions and orders the input DataFrame, scans it with the
compiled automaton under the configured skip policy, computes the
requested measures for every match, and assembles the results into a
DataFrame that mirrors standard SQL output---bringing declarative
pattern matching into conventional Python data-analysis workflows.
This chapter details the parsing, validation, and logical-planning
stages; Chapter~\ref{chap:implementation} continues with automata
compilation, Pandas execution, and optimization.

\section{SQL Parsing}
\label{sec:meth-compiler}
% ----------------------------------------------------------------

Transforming a \texttt{MATCH\_RECOGNIZE} query from plain text into
an executable representation requires a parsing pipeline that
handles both standard SQL syntax and the extended row-pattern
sub-language.  This section states the parsing requirements,
motivates the choice of parser technology, and describes the
resulting parser.

% ................................
\subsection{Parsing Approach and Existing Solutions}
\label{subsec:meth-what-parsing}
\label{subsec:meth-existing-parsers}
\label{subsec:meth-grammars}
% ................................

\emph{Parsing} analyzes an input string according to the rules of a
formal grammar and transforms it into a machine-processable
representation~\cite{mogensen2024compiler}.  The engine follows the
classical two-stage design~\cite{DragonBook}: a \emph{lexer}
segments the character stream into tokens (keywords, identifiers,
literals, punctuation), and a \emph{parser} matches the token stream
against context-free grammar rules, producing a parse tree for valid
input and a positioned error for invalid input.  Grammars are
written in EBNF, whose production rules of the form $N \to \alpha$
name each syntactic construct; SQL is a large context-free language,
and the \texttt{MATCH\_RECOGNIZE} sub-language adds productions for
pattern terms, quantifiers, alternation, grouping, anchors,
exclusions, \texttt{PERMUTE}, and the eight sub-clauses.

No existing Python SQL parser covers this sub-language adequately.
\texttt{sqlparse}~\cite{sqlparse_tool} produces only a shallow token
stream and does not model \texttt{MATCH\_RECOGNIZE};
\texttt{sqlglot}~\cite{sqlglot_tool} exposes a typed AST but lacks
project-ready coverage of the row-pattern constructs (exclusions,
anchors, \texttt{PERMUTE}, bounded quantifiers);
\texttt{mo\_sql\_parsing}~\cite{mo_sql_parsing_tool} converts SQL to
JSON with the same gap; and \texttt{pglast}~\cite{pglast_tool}, the
Python wrapper of the extracted PostgreSQL parser
(\texttt{libpg\_query}~\cite{libpg_query_tool}), cannot parse
\texttt{MATCH\_RECOGNIZE} at all.  A grammar-based parser generated
from a dialect that already includes the row-pattern productions was
therefore selected instead.

% ................................
\subsection{Parsing Requirements}
\label{subsec:meth-parsing-needs}
% ................................

This section examines what must be parsed and what the output
should look like.  Listing~\ref{lst:mr-full-syntax} shows the
supported syntax of a query containing a \texttt{MATCH\_RECOGNIZE}
clause in this implementation.

\begin{lstlisting}[
    language   = {},
    caption    = {Syntax of a \texttt{MATCH\_RECOGNIZE} query},
    label      = {lst:mr-full-syntax},
    basicstyle = \ttfamily\footnotesize
]
SELECT <expression> [, ...]
FROM   <table>
MATCH_RECOGNIZE (
  [ PARTITION BY <column> [, ...] ]
  [ ORDER BY <column> [ASC|DESC] [, ...] ]
  [ MEASURES <expression> AS <alias> [, ...] ]
  [ ONE ROW PER MATCH
  | ALL ROWS PER MATCH
    [ SHOW EMPTY MATCHES
    | OMIT EMPTY MATCHES
    | WITH UNMATCHED ROWS ] ]
  [ AFTER MATCH SKIP
    { PAST LAST ROW
    | TO NEXT ROW
    | TO FIRST <variable>
    | TO LAST  <variable> } ]
  PATTERN ( <row_pattern> )
  [ SUBSET <subset_name> = ( <variable> [, ...] ) [, ...] ]
  [ DEFINE <variable> AS <condition> [, ...] ]
) [ AS <alias> ]
<rest>;
\end{lstlisting}

The \texttt{DEFINE} clause is optional: a pattern variable that
appears in \texttt{PATTERN} without a predicate receives an implicit
always-true predicate in the logical plan.  Measure aliases require
the \texttt{AS} keyword in the adopted Trino grammar, while the
alias after the \texttt{MATCH\_RECOGNIZE} clause itself is optional.
The implementation focuses on one \texttt{MATCH\_RECOGNIZE} clause
extracted from a top-level \texttt{SELECT} query with an optional
outer \texttt{ORDER~BY}; full CTE rewriting, set-operation planning,
and general nested-subquery execution are outside the main execution
path, which is sufficient for the \texttt{FROM}-clause row-pattern
use cases evaluated in this thesis.  The parser must extract the
mandatory pattern clause and the optional
sub-clauses---\texttt{PARTITION~BY}, \texttt{ORDER~BY},
\texttt{MEASURES}, rows-per-match, after-match-skip,
\texttt{SUBSET}, and \texttt{DEFINE}---and deliver them through a
well-defined AST interface, described next.

% ................................
\subsubsection{The \texttt{MATCH\_RECOGNIZE} AST Interface}
\label{subsubsec:meth-ast-interface}
% ................................

The AST exposed by the parser to downstream stages is defined by a
hierarchy of typed Python \texttt{@dataclass} objects.  The root
node, \texttt{FullQueryAST}, holds the outer \texttt{SelectClause},
\texttt{FromClause}, and a single \texttt{MatchRecognizeClause}
object that groups the parsed row-pattern sub-clauses.  The interface
contract is:

\begin{itemize}
  \item Each node is represented as a typed Python dataclass or
    structured object.  Some metadata fields, especially those on
    \texttt{PatternClause}, are normalized after construction so that
    defined variables, implicit variables, and subset components can
    be resolved consistently.
  \item Each \texttt{MATCH\_RECOGNIZE} sub-clause maps to a
    dedicated typed dataclass
    (\texttt{PartitionByClause}, \texttt{OrderByClause},
     \texttt{MeasuresClause}, \texttt{RowsPerMatchClause},
     \texttt{AfterMatchSkipClause}, \texttt{PatternClause},
     \texttt{SubsetClause}, \texttt{DefineClause}),
    enabling the NFA compiler to dispatch on type rather than
    parsing raw strings.
  \item \texttt{DEFINE} predicates and \texttt{MEASURES}
    expressions are captured as expression strings during the first
    pass and interpreted on demand through a controlled expression
    evaluator (Section~\ref{subsec:meth-safe-eval}), keeping the main
    grammar concise.
  \item \texttt{Measure} nodes carry computed flags
    (\texttt{is\_classifier}, \texttt{is\_match\_number}) that
    allow the execution layer to route output columns without
    additional string inspection.
\end{itemize}

\subsection{Parser Technology Selection}
\label{subsec:meth-parser-generators}
\label{subsec:meth-alternatives}
\label{subsec:meth-antlr4}
% ................................

A \emph{parser generator}~\cite{mogensen2024compiler} takes a
grammar as input and produces the source code of a parser for it;
this automates lexer and parser construction and keeps the language
specification in one readable file.  Two non-generator approaches
were considered and rejected: a hand-written recursive-descent
parser is error-prone to maintain across SQL's hundreds of
productions, and regular expressions cannot model recursive
structures such as nested grouping, so they cannot serve as a
complete parser.  Three generators that target Python were compared:
\textbf{PLY} (Python Lex-Yacc)~\cite{ply_tool},
\textbf{Lark}~\cite{lark_tool}, and
\textbf{ANTLR4}~\cite{Parr2013}
(Table~\ref{tab:parser-gen-comparison}).

  \centering
  \small

\textbf{ANTLR4} (ANother Tool for Language Recognition,
version~4)~\cite{Parr2013,parr2014adaptive,antlr_website,parr_antlr_docs}
was selected: it produces top-down, Adaptive LL(*) recursive-descent
parsers from a \texttt{.g4} grammar file together with generated
listener/visitor scaffolding, which eliminates the manual traversal
boilerplate otherwise needed to build the typed AST of
Section~\ref{subsubsec:meth-ast-interface}; its parse-tree
visualiser aids grammar debugging; and an empirical study reports
higher performance and maintainability than
Lex/Yacc~\cite{antlr_performance}.

\subsubsection{Designing the Grammar}
\label{subsubsec:meth-grammar-design}
% ................................

The implementation uses the Trino SQL grammar obtained from the
ANTLR community grammars project
(\texttt{TrinoLexer.g4} / \texttt{TrinoParser.g4},
approximately 1\,521 lines across the two grammar files in this
project)~\cite{antlr_trino_sql}.
This grammar natively contains the \texttt{patternRecognition} rule
that covers all eight \texttt{MATCH\_RECOGNIZE} sub-clauses; no
modifications to the grammar file were required.
Listing~\ref{lst:grammar} shows the key productions.

\begin{lstlisting}[
    language = {},
    caption  = {Key productions of the \texttt{patternRecognition} rule in \texttt{TrinoParser.g4}},
    label    = {lst:grammar},
    basicstyle = \ttfamily\footnotesize
]
patternRecognition
    : aliasedRelation (
        MATCH_RECOGNIZE '('
          (PARTITION BY expression (',' expression)*)?
          (ORDER BY sortItem (',' sortItem)*)?
          (MEASURES measureDefinition (',' measureDefinition)*)?
          rowsPerMatch?
          (AFTER MATCH skipTo)?
          (INITIAL | SEEK)?
          PATTERN '(' rowPattern ')'
          (SUBSET subsetDefinition (',' subsetDefinition)*)?
          (DEFINE variableDefinition (',' variableDefinition)*)?
        ')' (AS? identifier columnAliases?)?
      )?
    ;

measureDefinition : expression AS identifier ;

rowsPerMatch
    : ONE ROW PER MATCH
    | ALL ROWS PER MATCH emptyMatchHandling?
    ;

skipTo
    : SKIP TO NEXT ROW
    | SKIP TO (FIRST | LAST)? identifier
    | SKIP PAST LAST ROW
    ;

rowPattern
    : rowPattern rowPattern             # patternConcatenation
    | rowPattern '|' rowPattern         # patternAlternation
    | patternPrimary patternQuantifier? # quantifiedPrimary
    ;

patternPrimary
    : identifier                                        # patternVariable
    | '(' rowPattern ')'                                # groupedPattern
    | '{-' rowPattern '-}'                              # excludedPattern
    | PERMUTE '(' rowPattern (',' rowPattern)* ')'      # patternPermutation
    ;
\end{lstlisting}

% ................................
\subsubsection{Handling Expressions as Strings}
\label{subsubsec:meth-expr-strings}
% ................................

\texttt{DEFINE} predicates and \texttt{MEASURES} expressions contain
column references, navigation function calls, arithmetic, comparisons,
and selected scalar or aggregate functions.  Parsing every expression
into a fully typed SQL expression tree would require a much larger
expression subsystem.  The adopted strategy is therefore to preserve
the relevant expression text during parsing and interpret it later
through the controlled expression evaluators used by the matching and
measure layers.  This two-pass strategy keeps the main grammar concise
while still avoiding direct execution of raw user expressions (see
Section~\ref{subsec:meth-safe-eval}).

% ................................
\subsubsection{Parse Tree vs.\ Abstract Syntax Tree}
\label{subsubsec:meth-ast}
% ................................

Parse trees produced by ANTLR4 contain many structural nodes
(keywords, punctuation) that are unnecessary for execution.  A
visitor traverses the parse tree and produces a compact
\emph{Abstract Syntax Tree} (AST) whose node types correspond
directly to semantic constructs~\cite{DragonBook,jones2003ast}.  The key node
types are summarised in Table~\ref{tab:meth-ast-nodes}.

\begin{table}[htbp]
  \centering
  \caption{Principal AST node types}
  \label{tab:meth-ast-nodes}
  \begin{tabular}{ll}
    \toprule
    \textbf{Node Type}                  & \textbf{Represents} \\
    \midrule
    \texttt{FullQueryAST}               & Root of the full query (SELECT + FROM + MR) \\
    \texttt{MatchRecognizeClause}        & Root of the \texttt{MATCH\_RECOGNIZE} sub-clause \\
    \texttt{PartitionByClause}           & \texttt{PARTITION BY} column list \\
    \texttt{OrderByClause}              & \texttt{ORDER BY} sort items (\texttt{SortItem}) \\
    \texttt{MeasuresClause}             & \texttt{MEASURES} list (\texttt{Measure} nodes) \\
    \texttt{RowsPerMatchClause}          & \texttt{ONE / ALL ROWS PER MATCH} mode \\
    \texttt{AfterMatchSkipClause}        & \texttt{AFTER MATCH SKIP} policy \\
    \texttt{PatternClause}              & \texttt{PATTERN} string with tokenizer \\
    \texttt{SubsetClause}               & \texttt{SUBSET} variable definitions \\
    \texttt{DefineClause}               & \texttt{DEFINE} list (\texttt{Define} nodes) \\
    \bottomrule
  \end{tabular}
\end{table}

The AST is validated for semantic correctness before pattern
compilation: unresolved pattern-variable references, invalid subset
definitions, incompatible output modes, and illegal pattern syntax are
detected and reported at this stage.  Pattern variables without an
explicit \texttt{DEFINE} predicate remain valid and are assigned an
implicit always-true predicate downstream.

\subsection{Visitor Implementation}
\label{subsec:meth-visitor}
% ................................

ANTLR4 generates an abstract visitor base class from the
grammar~\cite{robson2022designpatterns}; the parsing module builds
two cooperating visitors on it.  The outer
\texttt{FullQueryExtractor} handles the top-level statement rule,
extracts the outer \texttt{SELECT} list, \texttt{FROM} reference,
and trailing \texttt{ORDER~BY}, and locates the pattern-recognition
node.  It delegates to the inner \texttt{MatchRecognizeExtractor},
which extracts the sub-clauses in grammar order---partitioning,
ordering, measures with their \texttt{RUNNING}/\texttt{FINAL}
prefixes and aliases, rows-per-match mode, skip policy, the raw
pattern string, \texttt{SUBSET} definitions, and \texttt{DEFINE}
pairs---and then applies the validation routines that check required
clauses, balanced pattern syntax, incompatible output-mode
combinations, skip targets, and malformed navigation or aggregate
function usage.  The public entry point,
\texttt{parse\_full\_query(query)} $\to$ \texttt{FullQueryAST},
preprocesses the query string, drives the generated lexer and
parser, and replaces the default error listeners with a
\texttt{CustomErrorListener} that raises a structured
\texttt{ParserError} carrying the exact line, column, and source
snippet of any syntax violation.

\subsection{Testing and Evaluation of the Parser}
\label{subsec:meth-parser-testing}
% ................................

The parser is validated through the structured test suite in
\texttt{tests/}, organized into three layers.  Parser-focused tests
exercise the extraction and validation of every sub-clause,
including adversarial inputs: an undefined pattern-variable
reference, a prohibited function form, an invalid skip target, or an
incompatible output-mode combination triggers a
\texttt{ParserError} or an empty-result path as appropriate, while a
pattern variable without a \texttt{DEFINE} predicate remains valid
with an implicit always-true condition.  Integration tests adapted
from Trino's \texttt{TestRowPatternMatching.java} cover end-to-end
scenarios---quantifiers, exclusions, skip modes, output modes,
\texttt{CLASSIFIER()}/\texttt{MATCH\_NUMBER()}, navigation,
partitioning, \texttt{SUBSET} aliasing, and
\texttt{RUNNING}/\texttt{FINAL} semantics.  Dedicated files cover
SQL:2016 compliance details, empty-match handling, \texttt{IN~()}
preprocessing, \texttt{IS~NULL}, anchors, \texttt{PERMUTE},
back-references, pattern-cache thread safety, and aggregate
behavior.

\subsection{Discussion and Limits}
\label{subsec:meth-parser-discussion}
\label{subsubsec:meth-parser-limits}
% ................................

The grammar-driven approach keeps the language specification in a
single readable \texttt{.g4} file, provides structured parse-tree
generation with positioned syntax errors, and eliminates manual
traversal boilerplate through generated visitor scaffolding.  Two
limitations are known: advanced SQL constructs outside the supported
row-pattern subset (such as recursive CTEs inside predicate
subqueries) are not part of the main execution path, and because the
grammar follows Trino's dialect, vendor-specific extensions of other
systems may require minor grammar adjustments for cross-dialect
compatibility.

\subsection{Safe SQL Expression Evaluation}
\label{subsec:meth-safe-eval}
% ................................

\texttt{DEFINE} predicates and \texttt{MEASURES} expressions are
user-supplied strings that must be interpreted at runtime.  Directly
calling \texttt{eval()} on arbitrary user input would create a
code-injection vulnerability.  The implementation instead translates
the supported SQL expression subset into controlled Python expression
forms and evaluates them through AST-aware evaluators with explicit
operator and function handling.  Listing~\ref{lst:safe-eval} shows
the simplified principle.

\begin{lstlisting}[
    language = Python,
    caption  = {Safe SQL expression evaluation using AST whitelisting},
    label    = {lst:safe-eval}
]
import ast

def safe_evaluate(sql_expr, row_data, context):
    # 1. Translate SQL syntax to Python equivalents
    py_expr = translate_sql_to_python(sql_expr)

    # 2. Parse into a Python AST (expression mode only)
    tree = ast.parse(py_expr, mode='eval')

    # 3. Whitelist-validate: allow comparisons, arithmetic,
    #    boolean ops, literals, subscripts -- forbid imports,
    #    function calls (except navigation functions), assignments
    ASTSecurityValidator().visit(tree)

    # 4. Evaluate by walking only the approved AST nodes
    evaluator = ControlledExpressionEvaluator(row_data, context)
    return evaluator.visit(tree.body)
\end{lstlisting}

The runtime evaluator follows the same principle but uses the
project's \texttt{ConditionEvaluator} and measure-evaluation helpers:
it walks supported AST nodes explicitly and exposes only the current
row context, pattern-variable bindings, supported navigation
functions (\texttt{PREV}, \texttt{NEXT}, \texttt{FIRST},
\texttt{LAST}), and selected scalar or aggregate operations.  It does
not expose Python built-ins, imports, file-system access, network
access, or interpreter internals.

\section{Semantic Validation}
\label{sec:meth-semantic}
% ----------------------------------------------------------------

After the AST is constructed, the engine performs a systematic
semantic validation pass before any automaton is built.  This pass
catches errors that are syntactically valid but logically
inconsistent with the supported SQL:2016 row-pattern subset, and
applies a set of normalization rules that simplify downstream
processing.

% ................................
\subsection{Schema and Variable Consistency Checks}
\label{subsec:sem-schema}
% ................................

The validation and execution setup verify the following properties,
raising parser or execution errors when a query cannot be mapped
safely to the input DataFrame:

\begin{itemize}
  \item Columns referenced in \texttt{PARTITION BY},
    \texttt{ORDER BY}, \texttt{DEFINE}, and \texttt{MEASURES}
    expressions must be resolvable against the input DataFrame during
    execution setup or expression evaluation.

  \item Pattern-variable references in \texttt{MEASURES} or
    \texttt{DEFINE} expressions must refer to variables declared in
    the \texttt{PATTERN} clause or made available through
    \texttt{SUBSET}.  Variables without explicit \texttt{DEFINE}
    predicates are allowed and are treated as implicit always-true
    variables.

  \item If a \texttt{SUBSET} clause is present, the subset mapping is
    normalized and made available to later stages.  References to
    subset aliases and their component variables are then resolved
    where they are used in measures, predicates, or automaton
    construction.

  \item The \texttt{AFTER MATCH SKIP TO FIRST/LAST $V$} options
    reference a pattern variable or declared \texttt{SUBSET} alias
    $V$, catching typos at compile time rather than at runtime.
\end{itemize}

% ................................
\subsection{Navigation-Function Validation}
\label{subsec:sem-navfunc}
% ................................

Navigation functions impose inter-variable ordering constraints that
must be verified before the automaton is constructed.  The
\texttt{validate\_navigation\_conditions()} function in
\texttt{src/matcher/condition\_evaluator.py} performs this check by
building a \emph{variable timeline}: the order in which distinct
pattern variables first appear in the pattern string.  For each
\texttt{DEFINE} predicate the validator then checks:

\begin{itemize}
  \item Cross-variable \texttt{NEXT($v$.$c$)} references must point
    to a variable that appears later in the pattern timeline;
    self-references and unqualified navigation are resolved from the
    row context.  A navigation condition that is invalid for the
    pattern yields an empty result set rather than an exception,
    matching the Trino-oriented behavior used in the tests.

  \item \texttt{PREV($v$.$c$, $k$)} references are accepted when the
    referenced variable is available in the pattern context, with
    boundary conditions checked during execution.

  \item Nested navigation expressions such as
    \texttt{NEXT(PREV(A.price, 1), 1)} are handled by the enhanced
    navigation-evaluation path.
\end{itemize}

% ................................
\subsection{Output-Mode and Skip-Mode Consistency}
\label{subsec:sem-modes}
% ................................

Several combinations of output mode, skip mode, and pattern
features are invalid in the supported SQL:2016-oriented subset.  The
validator enforces the following rules:

\begin{itemize}
  \item Pattern exclusion syntax (\texttt{\{- \ldots -\}}) is
    incompatible with \texttt{ALL ROWS PER MATCH WITH UNMATCHED
    ROWS}; this combination is rejected with a descriptive error
    message.

  \item \texttt{AFTER MATCH SKIP TO FIRST/LAST $V$} accepts either
    a pattern variable or a \texttt{SUBSET} alias for $V$.  The
    validator resolves subset aliases against the declared
    \texttt{SUBSET} clause (an alias denotes the union of its member
    variables, so \texttt{SKIP TO $U$} skips to the last row matched
    by any member of $U$); a name that is neither a pattern variable
    nor a declared subset alias is rejected.  Skipping to a target
    that would re-anchor matching at the first row of the previous
    match is detected at runtime and reported as an error, since it
    would otherwise cause an infinite loop.
\end{itemize}

\noindent
After all checks pass, the validator records the resolved
configuration—normalized output mode, skip mode, implicit-variable
defaults, and measure semantics—ready for the logical planning step.

\section{Logical Plan Construction}
\label{sec:meth-logical}

canonical data structure that drives all subsequent phases without
further reference to the raw SQL text.  In the implementation this
materializes as a set of typed Python objects extracted from the

arguments to the execution functions.

\subsection{Implicit Variable Defaults}
\label{subsec:logical-implicit}
% ................................

SQL:2016 permits pattern variables to be listed in the
\texttt{PATTERN} clause without a corresponding entry in the
\texttt{DEFINE} clause.  Such variables match any row
unconditionally.  During logical plan construction the engine
identifies these implicit variables by comparing the set of names
extracted from the pattern tokens against the keys of the
\texttt{define} dictionary.  Any variable present in the former
but absent from the latter is assigned the always-true predicate
(represented as the Python string \texttt{"True"}), so that later
stages can treat all variables uniformly without special-casing
implicit ones.

\subsection{Measure Semantics Resolution}
\label{subsec:logical-measure-sem}
% ................................

Measure semantics (\textsc{Running} vs.\ \textsc{Final}) can be
specified explicitly using the keywords \texttt{RUNNING} and
\texttt{FINAL} in the \texttt{MEASURES} clause, or left implicit
for the engine to determine.  The resolution algorithm
(Figure~\ref{fig:measure-sem-resolution}) scans all measure
expressions in order:

\begin{enumerate}
  \item If any measure carries an explicit \texttt{RUNNING} or
    \texttt{FINAL} keyword, the query is in \emph{mixed-semantics
    mode} and all other measures without an explicit keyword
    default to \textsc{Final}.

  \item In single-semantics mode (no explicit keywords):
    \begin{itemize}
      \item All measures default to \textsc{Final} under
        \texttt{ONE ROW PER MATCH}.
      \item Under \texttt{ALL ROWS PER MATCH}: \texttt{LAST()}
        defaults to \textsc{Running}; \texttt{FIRST()},
        \texttt{PREV()}, and \texttt{NEXT()} default to
        \textsc{Final}; aggregate functions (\texttt{COUNT()},
        \texttt{SUM()}, \texttt{ARRAY\_AGG()}, and the
        multi-argument forms such as \texttt{MAX\_BY()},
        \texttt{MIN\_BY()}, and \texttt{STRING\_AGG()}) default to
        \textsc{Running}; all others default to \textsc{Final}.
    \end{itemize}
\end{enumerate}

\section{Chapter Summary}
\label{sec:meth-summary}
% ----------------------------------------------------------------

This chapter has presented the system design and methodology of the
proposed \texttt{MATCH\_RECOGNIZE} engine for Pandas DataFrames.  It
introduced the five-stage architecture---parsing, semantic
validation, logical plan construction, pattern compilation, and
execution---and explained how the first three stages transform a
user query into a validated logical representation.

The chapter first described the overall architecture and then focused
on the front-end methodology: SQL parsing, grammar selection,
ANTLR4-based parse-tree processing, AST construction, safe expression
handling, semantic validation, and logical-plan normalization.  These
stages establish the boundary between declarative SQL input and the
lower-level execution machinery.

Chapter~\ref{chap:implementation} continues from this logical design
by describing the automata compilation pipeline, the Pandas execution
model, the concrete implementation components, and the optimization
techniques used to improve runtime performance and memory behavior.
