\chapter{System Design and Methodology}
\label{chap:methodology}

This chapter describes how the engine turns a \texttt{MATCH\_RECOGNIZE}
query into something it can run directly on a
\texttt{pandas} DataFrame~\cite{monierashraf2025rowmatch}, and the
design decisions behind that pipeline.  It focuses on the front
end---the stages that take a query from raw text to a validated,
executable plan---while preserving the supported SQL:2016 row-pattern
semantics~\cite{iso2016}.  Chapter~\ref{chap:implementation} continues
from there into automata compilation, execution, and optimization.

% ----------------------------------------------------------------
\section{Architecture Overview}
\label{sec:meth-overview}
% ----------------------------------------------------------------

The engine is organized as five stages, each producing a well-defined
artifact that the next stage consumes
(Figure~\ref{fig:system-architecture}).  \emph{Parsing} turns the SQL
text into an abstract syntax tree (AST) that captures both the outer
\texttt{SELECT}/\texttt{FROM} and the \texttt{MATCH\_RECOGNIZE}
sub-clauses.  \emph{Semantic validation} checks that the query is
actually meaningful---variables, predicates, output modes, and skip
policies---before any work is committed.  \emph{Logical-plan
construction} flattens the validated AST into a single canonical
structure.  \emph{Pattern compilation} turns the pattern into a finite
automaton, and \emph{execution} runs that automaton over the ordered
data and returns a DataFrame.

Behind these stages sit three main runtime components.  The
\emph{parser} recognizes the query's parts---partitioning, ordering,
the pattern, its conditions, and the output measures---and organizes
them into the AST.  The \emph{automata compiler} converts the pattern
into a non-deterministic automaton and then a deterministic one, so
that repetition, alternation, and sequencing are all handled
uniformly~\cite{georgiou2025streaming}.  The \emph{execution engine}
partitions and orders the DataFrame, scans it with the compiled
automaton under the configured skip policy, computes each match's
measures, and assembles a result DataFrame that mirrors ordinary SQL
output.  The rest of this chapter covers the first three stages;
Sections~\ref{sec:meth-compiler}--\ref{sec:meth-logical} take them in
turn.

\section{SQL Parsing}
\label{sec:meth-compiler}
% ----------------------------------------------------------------

A \texttt{MATCH\_RECOGNIZE} query mixes ordinary SQL with an extended
row-pattern sub-language---quantifiers, alternation, grouping,
anchors, exclusions, \texttt{PERMUTE}, and eight sub-clauses---so the
parser must understand both.  Listing~\ref{lst:mr-full-syntax}
summarises the syntax the engine supports.

\begin{lstlisting}[
    language   = {},
    caption    = {Syntax of a \texttt{MATCH\_RECOGNIZE} query},
    label      = {lst:mr-full-syntax},
    basicstyle = \ttfamily\footnotesize
]
SELECT <expression> [, ...]
FROM   <table>
MATCH_RECOGNIZE (
  [ PARTITION BY <column> [, ...] ]
  [ ORDER BY <column> [ASC|DESC] [, ...] ]
  [ MEASURES <expression> AS <alias> [, ...] ]
  [ ONE ROW PER MATCH
  | ALL ROWS PER MATCH
    [ SHOW EMPTY MATCHES
    | OMIT EMPTY MATCHES
    | WITH UNMATCHED ROWS ] ]
  [ AFTER MATCH SKIP
    { PAST LAST ROW
    | TO NEXT ROW
    | TO FIRST <variable>
    | TO LAST  <variable> } ]
  PATTERN ( <row_pattern> )
  [ SUBSET <subset_name> = ( <variable> [, ...] ) [, ...] ]
  [ DEFINE <variable> AS <condition> [, ...] ]
) [ AS <alias> ]
<rest>;
\end{lstlisting}

The engine targets one \texttt{MATCH\_RECOGNIZE} clause inside a
top-level \texttt{SELECT} with an optional outer \texttt{ORDER~BY};
the \texttt{DEFINE} clause is optional, and any pattern variable used
without a predicate is treated as always-true.  Full CTE rewriting,
set-operation planning, and general nested subqueries lie outside this
path, which is sufficient for the \texttt{FROM}-clause row-pattern
cases evaluated in this thesis.

\paragraph{Choosing a parser.}
No existing Python SQL library parses this sub-language adequately:
\texttt{sqlparse}~\cite{sqlparse_tool} produces only a shallow token
stream, \texttt{sqlglot}~\cite{sqlglot_tool} and
\texttt{mo\_sql\_parsing}~\cite{mo_sql_parsing_tool} expose a typed AST
but omit constructs such as exclusions, anchors, \texttt{PERMUTE}, and
bounded quantifiers, and \texttt{pglast}~\cite{pglast_tool} cannot
parse \texttt{MATCH\_RECOGNIZE} at all.  Rather than extend one of
these, the engine reuses the community ANTLR4 grammar for the Trino
SQL dialect~\cite{antlr_trino_sql,Parr2013}, whose \texttt{.g4} files
already contain a \texttt{pattern\-Recognition} rule covering all eight
sub-clauses.  ANTLR4 generates the lexer, parser, and traversal
scaffolding directly from that grammar, so no grammar changes were
needed, and syntax errors are reported with the exact line, column,
and source snippet.  A hand-written recursive-descent parser or a set
of regular expressions was rejected: the former is hard to maintain
across SQL's hundreds of productions, and the latter cannot model
recursive structures such as nested grouping.

\paragraph{From parse tree to a typed AST.}
ANTLR4's raw parse tree carries many nodes---keywords and
punctuation---that execution does not need.  A pair of cooperating
visitors walks it and produces a compact, typed AST: a root node
holding the outer \texttt{SELECT}/\texttt{FROM} and a single
\texttt{MatchRecognizeClause}, with every sub-clause represented as its
own typed dataclass (partitioning, ordering, measures, rows-per-match,
skip policy, pattern, subset, and define).  Later stages then dispatch
on node type instead of re-inspecting raw strings.  The predicates in
\texttt{DEFINE} and the expressions in \texttt{MEASURES} are kept as
text at this stage and interpreted on demand, which keeps the grammar
small.

\paragraph{Safe expression evaluation.}
Because those predicate and measure strings are user-supplied, the
engine never calls Python's \texttt{eval()} on them---that would be a
code-injection risk.  Instead it translates the supported SQL
expression subset into a restricted Python form and evaluates it by
walking only an approved set of AST nodes: comparisons, arithmetic,
boolean logic, literals, subscripts, and the navigation functions
\texttt{PREV}, \texttt{NEXT}, \texttt{FIRST}, and \texttt{LAST}.
Imports, arbitrary function calls, assignments, and any access to
built-ins, the file system, or the network are all disallowed, so only
the current row context and pattern-variable bindings are ever
exposed.

The parser is exercised by a dedicated test layer that includes
adversarial inputs---undefined variables, prohibited function forms,
invalid skip targets, and incompatible output-mode combinations---and
end-to-end scenarios adapted from Trino's row-pattern test suite.  Two
limits follow from the design: SQL features outside the row-pattern
subset (for example recursive CTEs inside predicate subqueries) are not
on the main execution path, and because the grammar tracks Trino's
dialect, other vendors' extensions may need small grammar adjustments.

\section{Semantic Validation}
\label{sec:meth-semantic}
% ----------------------------------------------------------------

Once the AST exists, a validation pass runs before any automaton is
built.  It rejects queries that parse but have no well-defined meaning
in the supported subset, and normalises a few things for later stages.
The checks fall into three groups.

\emph{References must resolve.}  Every column in \texttt{PARTITION BY},
\texttt{ORDER BY}, \texttt{DEFINE}, or \texttt{MEASURES} must exist in
the input, and every variable reference must name a \texttt{PATTERN}
variable or a declared \texttt{SUBSET} alias.  A variable used in the
pattern but never defined is legal---it becomes an implicit always-true
variable---and subset aliases are resolved to their members.

\emph{Navigation must be consistently ordered.}  The validator builds a
timeline of when each variable first appears and checks that
cross-variable \texttt{NEXT}/\texttt{PREV} references point a valid way;
nested forms like \texttt{NEXT(PREV(A.price, 1), 1)} are supported.  A
condition that can never hold yields an empty result, not an
exception---matching the reference engines.

\emph{Output and skip modes must be compatible.}  The exclusion syntax
\texttt{\{-\ \ldots\ -\}} clashes with \texttt{ALL ROWS PER MATCH WITH
UNMATCHED ROWS} and is rejected; \texttt{AFTER MATCH SKIP TO
FIRST/LAST~$V$} must target a real variable or alias; and a skip that
would re-anchor at the previous match's start is caught, since it would
loop forever.  When every check passes, the validator records the
resolved configuration---output mode, skip mode, implicit-variable
defaults, and measure semantics---ready for planning.

\section{Logical Plan Construction}
\label{sec:meth-logical}
% ................................................................

The logical plan flattens the validated AST into one canonical
structure that drives every later phase without re-reading the SQL.  It
is a set of typed attributes pulled from the
\texttt{MatchRecognizeClause} and passed explicitly to the execution
functions (Table~\ref{tab:logical-plan}).

\begin{table}[htbp]
  \centering
  \caption{Principal logical-plan attributes extracted from the
    validated AST}
  \label{tab:logical-plan}
  \begin{tabular}{llp{6.5cm}}
    \toprule
    \textbf{Attribute} & \textbf{Type} & \textbf{Semantics} \\
    \midrule
    \texttt{partition\_by}   & \texttt{List[str]}
      & Columns for DataFrame partitioning; empty means a single
        global partition \\[4pt]
    \texttt{order\_by}       & \texttt{List[str]}
      & Columns for the intra-partition sort order \\[4pt]
    \texttt{pattern\_text}   & \texttt{str}
      & Normalized pattern string forwarded to the tokenizer \\[4pt]
    \texttt{define}          & \texttt{Dict[str, str]}
      & Variable $\to$ predicate; implicit variables get the
        always-true predicate \texttt{"True"} \\[4pt]
    \texttt{subset\_dict}    & \texttt{Dict[str, List[str]]}
      & Subset alias $\to$ constituent variables \\[4pt]
    \texttt{measures}        & \texttt{Dict[str, str]}
      & Measure alias $\to$ expression string \\[4pt]
    \texttt{measure\_semantics} & \texttt{Dict[str, str]}
      & Measure alias $\to$ \texttt{"RUNNING"}/\texttt{"FINAL"}, per
        the rules in Figure~\ref{fig:measure-sem-resolution} \\[4pt]
    \texttt{rows\_per\_match} & \texttt{RowsPerMatch}
      & Output-row multiplicity \\[4pt]
    \texttt{skip\_mode}      & \texttt{SkipMode}
      & After-match skip strategy \\[4pt]
    \texttt{skip\_var}       & \texttt{Optional[str]}
      & Target variable/alias for \textsc{To First}/\textsc{To Last} \\
    \bottomrule
  \end{tabular}
\end{table}

Two normalisations matter.  First, pattern variables with no
\texttt{DEFINE} get the always-true predicate \texttt{"True"}, so
downstream code treats every variable the same.  Second, each measure
is tagged \textsc{Running} or \textsc{Final}---from an explicit prefix,
or, when none is given, by the fixed precedence in
Figure~\ref{fig:measure-sem-resolution}.  The engine scans the measures
once to see whether any is explicit, then applies the rules top to
bottom: under \texttt{ONE ROW PER MATCH} all are \textsc{Final}; under
\texttt{ALL ROWS PER MATCH}, \texttt{FIRST}/\texttt{PREV}/\texttt{NEXT}
are \textsc{Final} and \texttt{LAST} \textsc{Running}, supported
aggregates default to \textsc{Running}, and anything else to
\textsc{Final}.

  \centering

    \node[rules, below=0.45cm of scan] (precedence) {\small

        \toprule
        \textbf{Priority} & \textbf{First condition that holds for a measure} &

        \midrule
        1 & Explicit \texttt{RUNNING} or \texttt{FINAL} prefix &

With these attributes resolved, the logical plan is assembled and
handed to the pattern-compilation phase of
Chapter~\ref{chap:implementation}.

\section{Chapter Summary}
\label{sec:meth-summary}
% ----------------------------------------------------------------

This chapter presented the front-end design of the proposed
\texttt{MATCH\_RECOGNIZE} engine: the five-stage architecture, and how
the first three stages---parsing, semantic validation, and logical-plan
construction---turn a user query into a validated logical
representation.  Along the way it explained why an ANTLR4 grammar for
the Trino dialect was reused rather than a new parser written, how
user expressions are evaluated safely, and how output and measure
semantics are resolved.  These stages mark the boundary between
declarative SQL input and the execution machinery that
Chapter~\ref{chap:implementation} takes up next, covering automata
compilation, the Pandas execution model, and the optimization
techniques behind the engine's runtime and memory behaviour.
